## 视频生成为什么比图像难一个数量级

### 计算量：从百万到十亿

先看数字：

```
单张 1024×1024 图片 = 3,145,728 像素值需要生成

一段 5 秒 24fps 1080p 视频：
  5 秒 × 24 帧/秒 = 120 帧
  每帧 1920×1080×3 = 6,220,800 像素值
  总计 = 120 × 6,220,800 = 746,496,000 像素值

  视频：图像 = 746M : 3.1M ≈ 240 倍
```

实际工程中，视频生成模型通过各种压缩策略来减轻负担：降分辨率（通常生成 512×512 而非 1080p）、时序降采样（不是 24fps 全帧率，而是先生成关键帧再插值）、潜在空间压缩（不在像素空间而在压缩后的潜在空间做扩散）。但即便高度优化后，一段 5 秒视频的生成仍然比一张图的生成慢 50-100 倍。

### 时序一致性：最难的那道坎

静态图片中你只需要画一只猫——视频中你需要这只猫从卧→坐→跳→走，且每一帧都必须是**同一只猫**。

时序一致性有三个层次：

**层次一：外观一致性**。毛发颜色会变吗？眼睛颜色会飘吗？体型的比例会在帧间漂移吗？这是最基础的一层，2024 年的模型基本解决了。

**层次二：物理一致性**。猫跳下桌子时，桌上的杯子会震动吗？如果不会——观众立刻出戏。如果会——模型的物理理解能力需要覆盖"猫的重力对桌面的影响通过桌腿传导到桌面上的杯子"这样的因果链。

**层次三：语义一致性**。"他从口袋里掏出一把钥匙，打开了门"——"掏"的动作必须有"手伸进口袋→取出物体→物体从口袋中被带出"的连续视觉逻辑。缺少任何一环，观众会感觉"动作不连贯"。

### 为什么 Sora 这么贵：算力账

Sora 关停时 OpenAI 透露日均运营成本约 1500 万美元。假设每天处理 10 万次生成请求：

```
单次生成成本 = $15,000,000 / 100,000 = $150

对比 DALL-E 3: 单张图片约 $0.05
视频生成成本是图像生成的 3000 倍
```

这不是技术做不到——是商业模式跑不通。消费者愿意为一段 10 秒的 AI 视频付多少钱？$5？$20？即便按 $20/次、每天 10 万次付费计算，日收入 $200 万，距日成本 $1500 万还差 7.5 倍。C 端市场目前没有足够的需求规模来支撑这种成本结构。

### 运动模糊和复杂动作

低光环境的视频中，快速移动的物体在每一帧上都是模糊拖影——这对图像生成模型来说是"噪音"，但对视频来说必须保留以符合物理直觉。

需要剧烈姿态变化的动作（跑步、跳舞、搏击）——身体各部位在相邻帧之间可能发生位置突变。视频模型必须"猜"出合理的中间态，而图像生成模型从未学过这种"猜运动"的能力。

## 技术路线：文生视频的三种方案

### 方案一：扩散模型的视频扩展

最主流的技术路线——把 2D 图像扩散模型扩展到 3D（空间+时间）。核心创新是在 U-Net 中插入**时间注意力层**：

```
标准图像扩散模型：
  噪声 → U-Net(2D卷积 + 自注意力) → 去噪 → 清晰图片

视频扩散模型：
  噪声 → U-Net(2D卷积+空间自注意力+时间自注意力) → 去噪 → 视频片段
                                    ↑
                          跨帧注意力：每一帧"看"其他帧，
                          确保运动一致性
```

代表模型：Stable Video Diffusion（Stability AI）、Modelscope（阿里达摩院）、AnimateDiff。这条路线的好处是能复用图像扩散模型的预训练权重——先用图像数据训练基础的"好看"能力，再加视频数据训练"连贯"能力。

### 方案二：扩散 Transformer（DiT）—— Sora 的路

Sora 的核心创新是把 U-Net 替换为 Diffusion Transformer（DiT）。它把视频在空间和时间两个维度上都切分成 Patch——类似于 ViT 对图像做 Patch 编码，然后交给 Transformer 处理：

```
一段视频 → 切成时空 Patch（空间×时间×时间块的立方体）
         → 展平成 Token 序列
         → Transformer 去噪
         → 重组为视频帧

视频理解中的 ViT Patch = 空间上的小块（如 16×16 像素）
视频生成中的 DiT Patch = 时空上的小块（如 16×16 像素 × 4 帧）
```

DiT 的优势是 Transformer 天然适合长距离依赖——不需要像 U-Net 那样通过多层卷积逐步扩大感受野。但对于高分辨率长视频，Token 数量会爆炸（这也是 Sora 成本失控的数学根源）。

### 方案三：自回归逐帧生成

另一种思路：先稳定扩散生成第一帧，然后**自回归地**——基于已有帧预测下一帧：

```
第1帧：扩散模型生成
第2帧：基于第1帧 + 运动向量生成
第3帧：基于第1,2帧 + 运动向量生成
...
```

这种方式的一致性较好（因为每一帧都以前面所有帧为条件），但容易产生误差累积——第 30 帧的微小形变会被第 60 帧放大。而且生成速度很慢——串行而非并行。

### 技术路线的现状对比

| 方案 | 一致性 | 速度 | 长视频 | 代表 |
|------|:--:|:--:|:--:|------|
| 视频扩散(U-Net) | ★★★☆☆ | ★★★★☆ | ★★☆☆☆ | SVD, AnimateDiff |
| 扩散Transformer(DiT) | ★★★★★ | ★★☆☆☆ | ★★★★☆ | Sora |
| 自回归逐帧 | ★★★★☆ | ★☆☆☆☆ | ★★★☆☆ | — |
| 混合方案(当前主流) | ★★★★☆ | ★★★☆☆ | ★★★☆☆ | Runway, 可灵 |

当前产品的技术路线大多是混合方案——融合视频扩散的效率和 DiT 的连贯性。

## Sora 的前世今生：一堂价值 1500 万美元的产品课

2024 年 2 月 15 日，OpenAI 发布了一段 60 秒的 AI 生成视频——东京街头的漫步，雨后的路面反光、行人的自然步伐、霓虹灯的反射。整个行业震惊了——这不是之前见过的"抽帧幻灯片"，而是真正的视频逻辑。

Sora 用了 Diffusion Transformer（DiT）架构，关键创新包括：

**时空 Patch 编码**。视频被同时在空间和时间维度上切分成 3D Patch（如 16×16 像素 × 4 帧），每个 Patch 包含了一小段时间内的局部视觉信息。这和 ViT 的图像 Patch 编码逻辑一致，但多了一个时间维度。

**原生可变分辨率和时长**。不像之前的模型固定在 256×256 或 512×512，Sora 能处理不同分辨率和不同时长的视频——因为 Transformer 的输入是可变长度的 Patch 序列。

**涌现的 3D 理解能力**。Sora 在没有显式 3D 建模训练的情况下，自发学会了摄像机运动时前景和背景的相对位移关系——这是一种"理解"。Transformer 从大量视频数据中归纳出了三维空间的隐含规则。

**但问题出在哪？**

Sora 的致命问题不是技术，是**产品-市场匹配**和**商业模式**：

问题一：Demo 惊艳 ≠ 用户能用。Sora 官宣的 demo 是精心挑选的——数百万次生成中筛选最佳片段。普通用户输入日常 Prompt 后，生成视频的可用率远低于预期。人多了 6 个手指、车在路面上漂移、影子反了方向——这些"不完美"在 1 秒的 gif 里不明显，在 10 秒的视频里立刻出戏。

问题二：范式错误。Sora 走的是"一键生成"路线——输入文字、出完整视频。但专业创作者的工作流不是这样——他们需要逐镜头控制、需要局部修改、需要保持角色一致性。Runway 走的"工具链"路线更符合实际需求——不是"AI 替你做"，而是"AI 帮你的创作工具"。

问题三：成本不可持续。日均 1500 万只够维持服务——不包括研发、不包括迭代、不包括获客。2026 年 4 月 26 日，Sora 消费者端正式关停。

Sora 给行业的最大教训：**最好的技术 ≠ 成功的产品。** 同时也意外地为 Runway 和国产方案清理了竞争赛道。

## Runway：从专业工具到创作者生态

Runway 不是 Sora 退出后才出现的——它从 2018 年就开始做 AI 视频工具。它的成功不是弯道超车，而是**专注专业创作者 6 年**的复利。

### 不只是一键生成，是完整的创作工作流

Runway 的产品哲学和 Sora 完全不同。Sora 追求"AI 自主创作"，Runway 追求"AI 增强人类创作者"：

```
Runway 的一个典型创作流程：

1. 文字/图片 → 生成视频片段（Gen-2/Gen-3 模型）
2. 不是一次性生成完——创作者逐段生成、逐段审核
3. 用关键帧控制工具指定片段的起始画面和结束画面
4. 运动笔刷（Motion Brush）：在画面上"画"出物体应该运动的方向和速度
5. 风格迁移：把实拍参考视频的艺术风格迁移到 AI 生成画面
6. 超分辨率升级：从 512p 提升到 1080p/4K
7. 色彩分级、音频同步、导出
```

每一步创作者都有控制权。这和 Midjourney 在图像领域的定位完全一致——不追求最先进的底层技术，但把产品工具链和创作者体验做到无可替代。

### 运动笔刷：Runway 最被低估的功能

Runway 的运动笔刷（Motion Brush）可能是被讨论最少但最实用的视频生成功能。它的工作原理：

1. 你上传一张静态图（或生成一帧）
2. 用笔刷在图上"画"出几个区域，每个区域指定运动方向
3. 指定哪些区域不动、哪些快速移动
4. Runway 生成视频时遵循这些运动指引

```
示例：一张城市街景的图

笔刷1：涂在天空区域 → 方向：向右 → 云朵向右飘移
笔刷2：涂在行人区域 → 方向：向前 → 行人向前走动
笔刷3：涂在建筑区域 → 静止 → 建筑保持不动
笔刷4：涂在树叶区域 → 方向：随机 → 树叶轻微摇摆
```

这解决了纯 Prompt 无法精确控制的"运动引导"问题。之前你只能写"云在动"，但运动笔刷让你指定"云的这一片以这个速度向这个方向移动"。

### Runway API 调用

```python
import requests

def generate_video_with_runway(prompt, image_url=None, seconds=5):
    """通过 Runway API 生成视频"""
    headers = {
        "Authorization": f"Bearer {RUNWAY_API_KEY}",
        "Content-Type": "application/json"
    }

    payload = {
        "model": "gen-3",
        "prompt": prompt,
        "duration": seconds,
        "resolution": "1280x768",
        "watermark": False
    }

    if image_url:
        payload["image_url"] = image_url  # 图生视频
        payload["motion_bucket_id"] = 127  # 运动强度 1-255

    # 提交生成任务
    response = requests.post(
        "https://api.runwayml.com/v1/video/generations",
        headers=headers, json=payload
    )
    task_id = response.json()["id"]

    # 轮询等待完成
    import time
    while True:
        status = requests.get(
            f"https://api.runwayml.com/v1/tasks/{task_id}",
            headers=headers
        ).json()

        if status["status"] == "SUCCEEDED":
            return status["output"]["url"]
        elif status["status"] == "FAILED":
            raise Exception(f"Generation failed: {status['error']}")

        time.sleep(3)

result_url = generate_video_with_runway(
    prompt="A cat jumping from a windowsill, slow motion, cinematic lighting, photorealistic",
    seconds=10
)
```

## 国产视频双雄：可灵与字节系

Sora 退场后，国产视频生成进入了可灵和字节系双雄对峙的局面。

### 可灵（快手）：物理感的胜出

可灵的技术路线和商业策略有几个关键选择：

**不做纯文生视频，主攻图生视频**。这个选择非常务实——从一张图开始生成，第一帧就是确定的。后面的运动基于确定的起点来预测，而不是从零开始同时预测内容和运动。一致性天然比纯文生视频好。

可灵的扩散过程：
```
输入：一张静态图片 + 文字描述想要的运动
    ↓
编码为潜在空间表示
    ↓
在时间维度上扩展——"预测如果这张图里的物体按照文字描述运动，后续帧应该是什么样"
    ↓
扩散去噪（保持第一帧不变，只对后续帧去噪）
    ↓
输出：起始帧=原图，后续帧=AI 生成的连贯运动
```

这个方案在运动物理感上比 Runway 的纯文生视频更稳定——因为起点确定了，不会出现"第一帧就画错了后面全跑偏"的雪崩效应。

**支持视频延长**。5 秒太短怎么办？把两段 5 秒连接起来——第一段的最后一帧作为第二段的起点。虽然不是最优雅的解法，但在实用层面有效解决了时长焦虑。

可灵的定价策略也值得关注：每日有免费生成次数（通常 3-6 次），付费比 Runway 便宜一个数量级。对个人创作者来说，这是目前门槛最低的高质量视频生成方案。

### Seedance（字节跳动）：人物一致性的突破

Seedance 的核心差异在于**人物外貌的跨视频一致性**。这对有"虚拟人设"的创作者来说极其重要——你希望你的 AI 角色在 100 段视频中看起来是同一个人。

Seedance 的技术路线：
```
第1次生成："一个穿着蓝色卫衣的年轻女孩走在街上"
    ↓
系统提取并保存人物的面部特征向量（"长相指纹"）
    ↓
第2次生成："同一个女孩在咖啡店里喝咖啡"
    ↓
注入第1次保存的特征向量作为生成条件
    ↓
输出：同一个女孩（脸型/发型/肤色一致）在新场景中
```

Seedance 不是在 Prompt 里写"保持和上次相同的角色"——它需要在嵌入空间中保存和复用"长相特征"。这是传统扩散模型架构的盲区——每次生成都是独立随机过程。

Seedance 的生态整合也是独有优势：豆包写脚本 → Seedance 生成视频片段 → 即梦剪辑 → 抖音发布。对短视频创作者来说，这条链路的切换成本为零——全在字节系内完成。

### 可灵和 Seedance 的选择

| 你的需求 | 选可灵 | 选 Seedance |
|---------|:--:|:--:|
| 追求运动自然度 | ✅ 更好 | — |
| 人物一致性要求高 | — | ✅ 独有优势 |
| 短视频批量创作 | — | ✅ 字节生态闭环 |
| 预算最低 | ✅ 都有免费额度 | ✅ 都有免费额度 |
| 图片转视频效果好 | ✅ 主攻方向 | — |
| 需要精确镜头控制 | ❌ 都不足 | ❌ 都不足 |

## 其他玩家：Veo、Pika、Luma

### Google Veo

Google 的视频生成模型，2026 年中仍处于有限开放状态（仅部分 Google Cloud 客户可申请）。核心特点：

- 与 Gemini 视频理解打通：先让 Veo 生成视频 → Gemini 分析是否满足要求 → 不合格则重新生成
- 画质和运动流畅度在早期测试中评价很高，但当前的有限开放策略限制了用户规模
- 长远看 Google 的最大优势是 YouTube——最大的视频训练数据源 + 天然的分发渠道

### Pika

Pika 在轻量级视频生成中独树一帜——它的特色不是"画质最好"，而是**"玩法最多"**：

- 口型同步（Lipsync）：上传一段音频 + 一张人脸图 → 生成对口型的说话视频
- 视频扩展（Expand）：在已有视频边缘向外扩展画面——有点像 Photoshop 的"内容感知填充"但是对视频的
- 画布编辑：在视频的特定帧上涂改，AI 自动平滑应用到相邻帧

Pika 的定位是"视频特效和趣味工具"，不是专业创作。它的免费额度非常充足，适合尝试和练习。

### Luma Dream Machine

Luma 的核心技术积累在 3D 重建领域——它的视频生成模型在**3D 视角一致性**上有独特优势：

- 摄像机环绕物体旋转时，物体不会在旋转过程中形变
- 多个视角之间的空间关系保持合理

这是因为 Luma 的训练数据包含大量的 3D 扫描和重建数据——模型在训练时学会了三维空间中的物体结构，而不是仅仅学会了二维外观。

## 当前技术天花板与避坑指南

2026 年中，AI 视频生成的真实能力天花板：

### 肯定能做到的

- **简单场景下的连贯运动**：人走路、车行驶、自然风光
- **风格化视频**：卡通、油画、赛博朋克风格的视频——因为风格化降低了"写实"的评判标准
- **图生视频比文生视频靠谱**：从确定的起始帧出发，一致性远好于纯文生

### 肯定还做不到的

**长时间视频**。10 秒左右的片段仍然是上限。通过拼接和延长可以做出更长的视频，但镜头切换和编排需要人工。一个 1 分钟的视频 = 6 段 10 秒片段 + 人工剪辑 + 转场设计。

**复杂动作形变**。跑步、跳舞、武术——身体的剧烈姿态变化仍然是裸眼可见的问题区域。手、脚在快速移动中最容易畸变——手指数量会变、脚会出现在不可能的位置。

**精确镜头控制**。目前的 Prompt 可以控制"内容"（什么在视频里），但很难控制"镜头"（怎么拍的）：

```
❌ 做不到：从低角度缓慢上摇的斯坦尼康跟拍，在猫跳起的瞬间切换到慢动作
✅ 能做到：一只猫从窗台跳下来，慢动作，电影感光照
```

Runway 的运动笔刷部分解决了这个问题，但离"精确到镜头参数"的控制还很远。

**口型同步（全覆盖）**。目前只有 Pika 等少数产品支持基础的口型同步。要让 AI 角色在说话时自然地张嘴、闭嘴、配合表情——这是下一阶段的核心需求，当前只有专项产品（HeyGen、Synthesia）在做。

### 避坑清单

- **不要期望一次就行**：平均每段满意视频需要 3-5 次生成尝试，文生视频的成功率更低
- **文生视频先从图生开始**：相同 Prompt，先生成一张满意的图片作为第一帧，再生成视频
- **背景不要太复杂**：多人、多运动物体、镜面反射——这些场景的失败率远高于简单场景
- **稳字优先**：使用 tripod/camera-stabilization 等关键词能降低画面抖动的概率

## 总结

- 视频生成从技术到产品经历了**三种方案**的演进：视频扩散 U-Net → 扩散 Transformer（Sora）→ 当前混合方案
- **Sora 关停是产品课而非技术失败**——日均 $1500 万成本对应每日 10 万次生成，C 端市场远未达到这个规模
- **Runway 胜在工具链**——不是最好的底层技术，但是最懂专业创作者需求的工具生态
- **可灵和 Seedance 通过差异化策略形成了国产双雄格局**——前者物理感（主攻图生视频）、后者人物一致性（跨视频保持长相）
- 当前天花板明确：10 秒左右、复杂动作不稳、镜头控制粗糙、口型同步只是起步
- **实用法则**：文生视频先从高质量图生开始，复杂场景降低期待，免费额度足够试错
- 下一篇看一个可能改写生成规则的新趋势：[原生多模态输出](./03-native-multimodal-output.md) —— 让同一个模型同时理解和生成
