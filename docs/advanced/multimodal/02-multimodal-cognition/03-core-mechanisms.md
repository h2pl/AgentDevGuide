## 三大机制全景

如果把多模态模型比作一位翻译官，那这三个机制就是他的核心能力：

```
表示（Representation） →  对齐（Alignment）  →  融合（Fusion）
   "看"                     "懂"                   "想"
把像素/波形/文字         学会"这张图和这段        综合所有信息做
变成模型内部的数学语言    话说的是同一件事"        出判断
```

三者的依赖关系是**顺序且不可逆的**——表示不当→对齐失败→融合混乱。一个具体例子：

```
输入：一张猫的图片 + "这是什么动物？"
预期输出："这是一只橘猫。"

表示阶段：图片 → 576个Patch向量，文字 → 5个Token向量
对齐阶段：模型知道"第127个Patch向量"和文字Token"猫"有关联
融合阶段：综合所有信息 → "橘猫"（颜色+品种）
```

如果表示阶段把图片压缩得太厉害（细节丢了），对齐就找不到"猫的耳朵"和文字"猫"的关联，融合就只能猜。

## 机制一：表示 — 把世界变成向量

神经网络只理解一种东西：**高维浮点数向量**。表示层的工作就是把人类世界的各种数据格式翻译成这种向量。

### 文本的表示：从字到向量

文本是多模态中最容易表示的模态，因为它已经是离散的符号序列。

**第一步：分词（Tokenization）**

BPE（Byte-Pair Encoding）分词器把文本切成子词单元：

```
输入："Transformers revolutionize natural language processing."
输出：["Transform", "ers", "Ġrevolution", "ize", "Ġnatural", "Ġlanguage", "Ġprocessing", "."]
```

为什么要切成子词？如果按完整单词切，词汇表会爆炸（英文有 100 万+ 单词）。如果切成单个字母，序列会太长。BPE 的平衡点是用 32000-100000 个子词覆盖几乎所有常见表达。这就像中文的偏旁部首——"𬞟"拆成"艹"和"频"，认得部首就能认得字。

**第二步：嵌入（Embedding）**

每个 Token 有一个唯一的整数 ID，通过一个可训练的查表矩阵（Embedding 矩阵）映射到高维向量：

```python
# 概念上等价于
embedding_matrix = nn.Embedding(vocab_size=32000, embedding_dim=4096)
token_vector = embedding_matrix[token_id]  # shape: [4096]

# 训练后，"猫"的向量和"狗"的向量距离很近（语义相似）
# "猫"的向量和"汽车"的向量距离很远（语义无关）
```

训练初期这些向量是随机的。经过海量文本训练后，语义相近的词在向量空间中的距离也相近。这是"表示"层最基础的语义理解。

**第三步：位置编码**

Transformer 是并行处理所有 Token 的——它没有天然的"顺序"概念。需要手动加入位置信息：

```python
# 位置编码把每个位置的索引变成一个向量
position_encoding = sin/cos(pos / 10000^(2i/d))  # 三角函数编码
final_token_embedding = token_embedding + position_encoding
```

### 图像的表示：ViT 如何"看"图

图像是连续的像素分布，不是离散符号。ViT 的任务是把连续像素变成离散的 Patch Token。

```python
# ViT 的核心流程（简化）
image = load_image("cat.jpg")                    # shape: [3, 224, 224]
patches = image.unfold_patches(patch_size=16)     # shape: [196, 768]
                                                   # 224/16 = 14, 14×14 = 196 patches
                                                   # 每个 patch: 16×16×3 = 768 个像素值

patch_embeddings = linear_projection(patches)      # shape: [196, 1024]
                                                   # 768 维像素向量 → 1024 维语义向量

# 加上 CLS Token 和位置编码
cls_token = learnable_cls_token                     # shape: [1, 1024]
position_embeddings = learnable_pos_embeddings       # shape: [197, 1024]

input_to_transformer = torch.cat([cls_token, patch_embeddings]) + position_embeddings
# shape: [197, 1024]  — 197个Token送入Transformer
```

**关键设计 1：Patch 的分辨率权衡**

Patch 越大 → Token 越少 → 计算越快 → 但细节损失越多。Patch 越小 → Token 越多 → 细节越好 → 但计算量平方增长。

```python
# 同样的 224×224 图片，不同 Patch 大小的效果
patch_size=32: 7×7=49个Token   → 很快，但只能识别大物体
patch_size=16: 14×14=196个Token → 平衡点（CLIP/LlaVA默认）
patch_size=8:  28×28=784个Token → 精细，但速度慢4倍
```

这是 ViT 最核心的工程权衡——决定多模态模型的"视力"有多好。

**关键设计 2：CLS Token**

Transformers 需要区分"代表整张图的全局理解"和"每个 Patch 的局部理解"。CLS Token 就是那个"全局代表"——它最初是一个随机初始化的可学习向量，在 12-24 层 Transformer 后，这个 Token 会吸收来自所有 Patch 的信息，最终形成对整张图的理解。

在做图文对比学习时，CLS Token 的输出向量就是对整张图的表示——CLIP 就是用这个向量和文本表示进行对比的。

### 音频的表示：从波形到频谱到 Token

音频处理的流水线比图像多一步——先把一维波形变成二维频谱图，再用类似 ViT 的方法处理。

```python
import torchaudio

# 原始音频
waveform, sample_rate = torchaudio.load("speech.wav")  # shape: [1, 160000]
                                                        # 10秒 × 16000Hz

# 梅尔频谱变换
mel_spectrogram = torchaudio.transforms.MelSpectrogram(
    sample_rate=16000,
    n_mels=128,      # 128个梅尔频带
    n_fft=400,       # 25ms 窗口
    hop_length=160   # 10ms 步长
)(waveform)
# shape: [128, 1000]  — 128个频带 × 1000个时间帧

# 然后切成Patch，和ViT一样的处理流程
# 128×1000的频谱图 → 切成若干时间频带patch → Token序列 → Transformer
```

**关键数值**：1 秒 16kHz 音频 = 16000 个采样点。经过梅尔变换后 = 128 个频带 × 100 个时间帧（每秒）。再经过 Patch 切分 = 约 250-500 个 Token（取决于 Patch 大小）。

**为什么需要梅尔变换**：原始波形包含太多高频噪声，直接编码效率极低。梅尔频谱模拟了人耳对频率的非线性感知——低频分辨得更细（人耳对低频变化敏感），高频更粗——在保留关键信息的同时压缩了数据量。

### 原生统一的 Token 化

GPT-4o 和 Gemini 的图片 Token 化与 CLIP ViT 有本质区别：

**CLIP ViT**：把图片切 Patch → 用 12 层 Transformer 编码成特征向量 → 这些特征向量再被"翻译"到 LLM 空间

**GPT-4o**：图片的像素区域直接被一个**与 LLM 共享参数的 Tokenizer**编码成 Token → 这些 Token 和文本 Token 是同一个嵌入空间的原生元素

区别在于：CLIP ViT 是"第三方翻译官"，GPT-4o 的 Tokenizer 是"母语者"。翻译官的翻译再好也会有失真，母语者不会。

**代价**：GPT-4o 处理一张高清图片可能产生数千甚至上万个 Token——远超 CLIP ViT 的 196 个 Token。更高的视觉精度换来更高的 Token 成本。

## 机制二：对齐 — 让图像和文字互相理解

表示让图片和文字都变成了向量。但"一只猫的图片"的向量和文字"猫"的向量，初始状态下**在数学上没有任何关联**。对齐机制的任务就是在向量空间中建立这种关联。

### 为什么对齐这么难

类比理解：假设你有两个人——一个只能说中文，一个只能说英文。你给了他们各一万本各自语言的书，让他们各自形成了对世界的理解。现在你想让他们交流——你需要一个翻译。

中文的理解空间和英文的理解空间是两套独立的坐标系统。中文里"猫"的位置，和英文里"cat"的位置，在没有训练之前没有任何对应关系。

图片和文字的对齐比这更难——因为图片不是语言。"一只橘猫趴在窗台上，午后阳光斜照"这句话对应的不是一个固定的像素排列，而是无数种可能的光影、姿态、构图的照片的集合。模型需要学会从这些变化中提取不变的本质。

### 方案一：对比学习 — CLIP 的方式

CLIP（Contrastive Language-Image Pre-training）是 2021 年 OpenAI 提出的对齐方案，至今仍是所有多模态模型的地基。

**训练数据**：4 亿对（图片，文字描述）。每对都是一张图片配上它的自然语言描述——来自互联网上自然存在的图文配对（网页图片的 alt text、社交媒体的图文帖子、电商产品的图和描述）。

**训练方式**：

```python
# 一批训练数据：N 张图片 + N 段对应文字
# 目标：正确的图文对的向量要接近，错误配对要远离

image_features = image_encoder(images)       # shape: [N, 512] — N张图的向量
text_features = text_encoder(texts)          # shape: [N, 512] — N段文字的向量

# 归一化
image_features = normalize(image_features)
text_features = normalize(text_features)

# 相似度矩阵：每一行是一张图，每一列是一段文字
similarity = image_features @ text_features.T  # shape: [N, N]

# 理想情况：对角线元素（正确配对）的相似度最高
# 训练目标：最大化对角线，最小化非对角线

loss = cross_entropy(similarity / temperature, labels=[0,1,2,...,N-1])
```

**关键设计：温度参数**

温度参数 temperature（通常 0.07）控制相似度分布的"尖锐程度"。温度越低，模型对"哪些算相似、哪些不算"的判断越严格。

```
高温（0.1）：模型说"这张图和这段文字相似度 0.62"
低温（0.01）：模型说"这张图和这段文字相似度 0.98，其余最高只有 0.05"
```

训练初期用较高温度（容忍模糊），后期降低温度（要求精确）。

**图文检索的直观理解**

训练 4 亿对图文后，CLIP 的向量空间内部形成了非常清晰的语义结构：

- "狗"的图片向量聚集在一个区域，"狗"的文字向量聚集在同一区域
- 用英文搜猫的图片：CLIP 找到"cat"文字向量附近的图片向量
- 用中文搜猫的图片：虽然 CLIP 主要训练英文，但它的多语言版本也能跨语言——因为"猫"和"cat"的语义在空间中非常接近

**CLIP 的局限**

1. **粒度太粗**：CLIP 的对齐是"整张图 vs 整段文字"。一张图里有猫、有沙发、有窗户，文字说"一只猫在沙发上"，CLIP 无法建立"猫"这个词和图中猫的区域之间的精细对应——它只知道"整体上这张图和这段文字是匹配的"
2. **偏自然图片**：训练数据以互联网上的自然照片为主，对文档、图表、UI 截图的理解偏弱——虽然 2026 年的 CLIP 变体（SigLIP）已经有所改进
3. **不支持推理链**：CLIP 只做匹配（match），不做推理（reason）。你问它"哪只猫比较胖"——CLIP 能告诉你两只猫都在图里，但不能比较

### 方案二：Q-Former — 精细化的视觉特征提取

Q-Former 解决了 CLIP 的最大痛点——"整张图 vs 整段文字"太粗糙。

**核心思想**：不要把所有图像信息一股脑丢给 LLM。先让一组"带着问题的查询向量"从图像中提取和问题最相关的信息。

```python
# Q-Former 的核心结构（简化）
class QFormer:
    def __init__(self):
        self.query_tokens = nn.Parameter(torch.randn(32, 768))  # 32个可学习查询向量
        self.cross_attention = CrossAttention()
        self.self_attention = SelfAttention()

    def forward(self, image_features, text_query=None):
        # 第一步：查询向量之间互相交流
        queries = self.self_attention(self.query_tokens)
        # "我们要回答什么问题？需要图像的哪些信息？"

        # 第二步：查询向量从图像特征中提取相关信息
        queries = self.cross_attention(queries, image_features)
        # "第1个查询向量：提取物体类别信息"
        # "第2个查询向量：提取空间关系信息"
        # "第3个查询向量：提取颜色和纹理信息"
        # ...

        # 第三步：再次互相交流，综合提取结果
        queries = self.self_attention(queries)

        return queries  # shape: [32, 768] — 32个精选的视觉Token
```

对比第一代 LLaVA 直接把 ViT 的 576 个 Patch Token 全部丢给 LLM，Q-Former 只有 32 个精心挑选的查询 Token——**大幅降低 LLM 的输入长度，同时保留了最关键的信息**。

**Q-Former 的两阶段训练**

- 第一阶段：图文匹配——让 Q-Former 学会"什么样的查询向量能提取出和文字最相关的视觉信息"
- 第二阶段：图文生成——让 Q-Former 学会用提取出的视觉信息辅助 LLM 生成回答

### 方案三：简单投影 — LLaVA 的极简主义

LLaVA 用一个矩阵乘法完成对齐：

```python
# LLaVA 投影层的全部逻辑
projection = nn.Linear(1024, 4096)  # 唯一需要训练的参数
visual_tokens = projection(vit_output)  # 768维→4096维，直接映射
```

为什么这么简单的方案能工作？因为这依赖于 CLIP 已经完成的巨大工作量——CLIP 已经把图像特征和文本特征对齐到了同一个语义空间。LLaVA 的投影层只是做**格式转换**——把 CLIP 的输出维度（1024）转成 LLaMA 的嵌入维度（4096）。

没有 CLIP，这个简单方案完全不可行。**LLaVA 证明了：在当前阶段，对齐工作应该由专门的对比学习任务完成，而不需要在 LLM 内部重复做。**

### 方案四：原生交叉注意力 — GPT-4o 的端到端对齐

前三代方案都是"显式对齐"——CLIP 对比学习、Q-Former 查询提取、LLaVA 线性投影，都有专门的对齐模块。

GPT-4o 的对齐是"隐式"的——没有专门的对比学习模块，没有投影层。**对齐能力是通过端到端的多模态预训练，自动在 Transformer 的自注意力层中形成的。**

```
GPT-4o 内部（某一次自注意力计算时）：

文字Token "猫" 的注意力权重分布：
  - 图片Token[42] (猫的头部区域)   权重: 0.28  ← 最高
  - 图片Token[43] (猫的身体区域)   权重: 0.18
  - 图片Token[89] (背景窗户)      权重: 0.01
  - 文字Token "橘"                 权重: 0.15
  - ...其余数千个Token

Transformer 自动学会了：
"当思考'猫'这个文字概念时，应该主要看图片中对应的物体区域"
```

这种对齐能力的形成不需要专门的对比学习任务——它是自注意力机制在大规模图文联合训练中的自然涌现。模型中没有任何地方写着"文字'猫'应该对齐图片中的猫区域"——它自己从数据中学到了。

## 机制三：融合 — 把多模态信息合在一起推理

对齐让模型知道了"哪些信息是相关的"。融合让模型**综合所有相关信息做出判断**。

### 融合策略对比

| 策略 | 融合时机 | 优点 | 缺点 |
|------|---------|------|------|
| **早期融合** | 最底层——原始Token时期就混在一起 | 信息最完整 | 计算量最大 |
| **中期融合** | Transformer中间层混合 | 平衡效率和信息 | 架构设计复杂 |
| **后期融合** | 各模态独立推理完，最后汇总 | 最简单 | 缺少跨模态交互 |

绝大多数现代多模态模型使用**中期融合**。

### 自注意力：融合的数学引擎

融合的真正引擎是 Transformer 的自注意力机制。拆开看它怎么在多模态场景下工作：

```python
# 自注意力的简化实现
def self_attention(query, key, value):
    # query, key, value 都来自同一个输入序列
    # 在多模态场景下，序列 = 图片Token + 分隔Token + 文字Token

    # 计算注意力分数：每个Token对其他所有Token的"关注度"
    scores = query @ key.T / sqrt(d_k)     # shape: [N, N]
    attention_weights = softmax(scores)       # 归一化为权重

    # 加权求和：每个Token的输出 = 所有Token的value的加权组合
    output = attention_weights @ value       # shape: [N, d_v]
    return output
```

在最早的一层 Transformer 中，Token 之间的关注还是局部的——文字 Token 关注邻近的文字 Token，图片 Patch 关注邻近的图片 Patch。但随着层数加深：

- 第 3-6 层：出现了跨模态关注——图片 Token 开始关注和它语义相关的文字 Token
- 第 12-18 层：跨模态关注高度成熟——任何一个 Token 都能精确定位它在另一个模态中的语义对应
- 第 24+ 层：形成了综合性的全局理解——模型不是在"看图+读文字"，而是在一个统一的语义空间里思考

### 一层的完整流程

```
输入层： [I₁][I₂]...[I₅₇₆][SEP][T₁][T₂]...[T₅₀]

第1层自注意力：
  I₁关注 → I₁(近邻Patch) + I₂(近邻) + I₃(近邻)  （图像内部）
  T₁关注 → T₂ + T₃                               （文本内部）

第6层自注意力：
  I₁关注 → I₁...I₅₇₆ + T₁ + T₂ + T₅              （开始跨模态）
  T₅关注 → T₁...T₅₀ + I₂₃ + I₄₅ + I₁₂₈           （文字开始关注图片）

第18层自注意力（深度融合）：
  I₂₃关注 → 全模态均匀分布                        （形成了全局理解）
  T₅关注 → 全模态均匀分布

第24层 → 输出：
  每个Token都包含了全模态的综合信息
  取最后一个Token的输出 → 送入输出层 → 生成回答
```

### 残差连接：防止信息丢失

Transformer 的每一层都有一条"短路"——残差连接：

```python
# 标准 Transformer 层
x = x + self_attention(layer_norm(x))     # 注意力 + 残差
x = x + feed_forward(layer_norm(x))       # 前向传播 + 残差
```

在多模态融合中，残差连接的价值在于：即使某个模态的 Token 在这一层没有得到足够的"关注"，它的原始信息通过残差直接传到下一层——不会因为一层忽略而永久丢失。这大大增强了融合的鲁棒性。

## 机制协同：一条完整链路的剖面

用你上传一张猫的照片问"这是什么品种的猫？"来走一遍三个机制的协同：

```
阶段 1：表示（<10ms，预处理）
  图片 → ViT → 576个 Patch Token（每个1024维）
  文字 → BPE → 12个 Token → Embedding → 12个文本向量（4096维）
  投影层 → 576个1024维 → 576个4096维（对齐到LLM空间）

阶段 2：拼接（<1ms）
  [I₁...I₅₇₆] + [SEP] + [T₁...T₁₂] → 589个Token输入

阶段 3：对齐 + 融合（主要计算，数百ms）
  Layer 1-6：各模态内部自注意力
  Layer 7-12：跨模态注意力建立 → I₁₂₈ 和 T₃("猫") 开始关联
  Layer 13-20：深度融合 → 所有Token互相理解
  Layer 21-32：高层推理 → 模型"知道"看到的是猫，且是橘猫

阶段 4：输出（逐Token生成）
  输出层 → "这" → "是" → "一" → "只" → "橘" → "猫" → "，"
         → "属" → "于" → "英" → "国" → "短" → "毛" → "猫" → "品" → "种" → "。"

每个生成步骤中，输出Token重新"看"一遍所有输入Token（包含图片）
——这意味着模型在说"英短"时，还在确认猫的脸型和毛色
```

整个流程从上传到第一个字输出，大约 0.5-2 秒。这背后是 32 层 Transformer、每个 Token 之间上千次交互、数十亿参数的协同工作。

## 这些机制如何影响你的选择

| 维度 | 如果你想... | 关键技术考量 |
|------|------------|-------------|
| 本地部署 | 在消费级显卡跑多模态 | 选第一代（LLaVA系），ViT Patch数越少越快 |
| OCR 精度 | 识别小字和细密文字 | 选高分辨率支持（LLaVA 1.6动态高分辨率 / Qwen-VL高分辨率模式） |
| 复杂推理 | 需要综合分析图片中的逻辑关系 | 选第三代（原生统一），跨模态注意力层数更多 |
| 中文场景 | 处理中文文档和中文图片 | 选 Qwen-VL，对齐训练包含中文图文数据 |
| 微调 | 在自己的业务数据上定制 | 选开源 + Q-Former或简单投影架构，对齐模块清晰 |
| 视频 | 需要时序理解 | 只能用 Gemini（原生视频支持），抽帧方案效果差 |
| 成本敏感 | Token 消耗要低 | 第一代模型（ViT Token少） + Q-Former（进一步压缩Token） |

## 总结

- 三大机制环环相扣：**表示**决定能看到多少信息，**对齐**决定能从信息中提取什么，**融合**决定能综合信息推理到什么程度
- **表示的核心是 Patch 大小（ViT 分辨率）和 Token 数量（成本）之间的权衡**
- **对齐有四种方案**：CLIP 全局对比（地基）、Q-Former 精细化提取、LLaVA 极简投影、GPT-4o 端到端隐式对齐——复杂度递增，能力也递增
- **融合的引擎是自注意力**——不是"先看图再读文"，是每一层 Transformer 中所有 Token 同时互相"看"，逐步加深跨模态理解
- **残差连接是融合的保险**——确保某个模态的信息不会因为一层没关注而永久丢失
- 02 章全部结束。你已经有了多模态的完整认知框架：技术定义→三代演进→三大机制。下一篇进入最丰富也最实用的专题——[03 视觉语言模型（VLM）](../03-vision-language-models/README.md)，从 CLIP 到 GPT-4o，看这条最成熟的技术线每一步是怎么走的
