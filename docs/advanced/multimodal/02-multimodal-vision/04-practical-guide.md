# 多模态视觉实战指南：从选型到落地

多模态视觉能力已经从"能不能看懂图片"进化到"能不能在真实业务中稳定产出"。这篇指南不讨论原理，只回答三个问题：怎么选、怎么用、怎么排查。

## 第一步：判断你的场景是否需要视觉

这是最常被忽略的问题。很多场景用文本模式就能解决，强上视觉只是增加成本和延迟。

| 输入类型 | 推荐方式 | 原因 |
|---------|---------|------|
| 扫描版 PDF（纯文字） | OCR + 文本模型 | 视觉模型对扫描 PDF 的 token 消耗比 OCR 高 10-20 倍 |
| 带表格的截图 | 多模态模型 | OCR 管线对复杂表格的还原率仅 60-70%，GPT-5.5/Gemini 可达 90%+ |
| 产品图片分类 | 专用分类模型 | 更便宜、更快，CLIP 嵌入后做 KNN 分类 |
| 图文混合的广告图 | 多模态模型 | 需要理解图文关系（"不要买这个"旁边画了个产品） |
| 实时视频监控 | 专用视觉模型 | 多模态模型 1-2s 延迟不适合实时场景 |

**黄金法则**：如果只用 OCR + 文本就能完成，别上视觉模型。如果涉及图文关系理解、视觉推理、空间关系判断，则必须用多模态模型。

## 第二步：模型选型决策树

```
需要 Visual Grounding（坐标定位）？
  ├─ 是 → Claude Opus 4.8（Grounding 最强）或 Qwen2.5-VL（开源替代）
  └─ 否 → 需要多语言/非英语？
       ├─ 是 → Gemini 3.1 Pro（多语言视觉理解领先）
       └─ 否 → 需要长视频/视频分析？
            ├─ 是 → Gemini 3.1 Pro（原生视频 token 支持）
            └─ 否 → 成本敏感？
                 ├─ 是 → Gemini 2.5 Flash / GPT-5-mini
                 └─ 否 → GPT-5.5 / Claude Opus 4.8
```

实测参考（2026-06）：
- **屏幕截图分析**（Computer Use / GUI Agent）：Claude Opus 4.8 > GPT-5.5 > Gemini 3.1 Pro。Claude 在按钮定位和界面元素识别上优势明显
- **文档分析**（复杂表格/图表）：GPT-5.5 ≈ Gemini 3.1 Pro > Claude Opus 4.8。两者对多页 PDF 的处理各有千秋
- **视频分析**（长视频摘要/时序推理）：Gemini 3.1 Pro > GPT-5.5 > Claude。Gemini 原生视频 token 处理在长视频场景遥遥领先
- **通用图像理解**（自然图片描述/QA）：三者差距很小。按价格选择即可

## 第三步：多步视觉推理的工作流设计

单次视觉推理能做的事情有限。大多数生产场景需要多步推理链：

```
输入图片 → 第一步：图片分类（"这是什么类型的文档？"）
         → 第二步：针对性提取（"提取表格中的金额列"）
         → 第三步：验证（"金额列是否包含非数字字符？"）
         → 输出结构化结果
```

**实践要点**：
- 给每步一个明确的输出约束（JSON Schema 或格式要求）
- 第二步的 Prompt 可以使用第一步的结果做上下文
- 关键步骤做交叉验证（两个模型分别跑一次，对比结果）
- 设计 fallback 路径——模型不确定时降级到规则方案

**工具链建议**：

```
轻量需求（<1000 次/天）：
  Python script + 直接调用 API + prompt 模板

中等需求（1000-50000 次/天）：
  LangChain/LlamaIndex 编排 + 缓存层 + 结果验证

高吞吐需求（>50000 次/天）：
  Batch API + 分布式处理 + 规则引擎兜底
```

## 第四步：图像质量与 Prompt 优化

图像质量直接影响模型理解准确率：

1. **分辨率**：并非越高越好。Gemini 默认 3072x3072 上限，超过会被压缩。GPT-5.5 支持细节模式（high res）但 token 翻 4 倍。通常 1024x1024 足够
2. **格式**：PNG 无损但文件大，JPEG Q90 对模型理解几乎无影响且体积小 10 倍。WebP 是折中方案
3. **预处理**：裁剪掉无关边框可大幅提升关键区域的理解准确率。图像去偏斜、对比度增强对文档 OCR 有明显帮助
4. **Prompt 技巧**：给模型指定"看哪里"——"请关注图像左上角的图标"比"描述这张图"准确率高 30-50%。要求结构化输出（JSON）比自由文本更稳定

## 第五步：监控与排障

多模态视觉应用比纯文本应用的故障点多两个维度：

| 故障类型 | 表现 | 排查方向 |
|---------|------|---------|
| 模型看不到内容 | 回答"图片中无文字"但实际有 | 检查图像分辨率是否达标、压缩是否过度、是否为支持的格式 |
| 理解不准确 | 识别出文字但理解语义错误 | 检查 Prompt 是否过于模糊、图像是否有多余元素干扰 |
| 定位偏移 | 说"左上角有个按钮"但实际在右边 | 检查图像是否被缩放/裁切、坐标系统是否一致 |
| 时序混乱 | 视频事件顺序说错 | 检查帧采样率是否过低、关键事件是否被跳帧 |

**核心指标**：
- 空响应率（模型说"看不清"的比例 > 5% 说明图像质量有问题）
- 结构化解析成功率（JSON 解析失败率 > 2% 说明输出格式不稳定）
- Token 消耗/次（异常突增说明图像大小失控或 Prompt 膨胀）
- 端到端延迟（P95 > 5s 需要优化 Pipeline）
