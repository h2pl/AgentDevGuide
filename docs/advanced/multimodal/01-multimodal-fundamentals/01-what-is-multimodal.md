# 什么是多模态

你每天处理的信息都是多模态的。刷朋友圈时看到一张美食照片（图像），点开后朋友发了一段语音说"这家超好吃"（音频），你回了一句"周末去试试"（文字）。打开抖音刷了条 AI 生成的视频（视频 + 音乐 + 文字），又顺手用 Midjourney 生成了一张海报（文生图）。

整个过程自然流畅，你甚至不会意识到自己在切换不同模态。但让 AI 做到同样的事，是过去十年最硬核的技术挑战之一。

2024 年之前，市面上没有一个模型能同时"看懂"图片、"听懂"语音、"理解"视频。你要做一个能处理图片的应用，需要一个视觉模型；要处理语音，需要接入 ASR 服务；要生成图片，需要调用文生图 API——每个能力是一个独立的产品，之间不共享上下文。多模态 AI 要解决的就是这个割裂：让一个模型能同时处理和理解多种类型的信息，并且理解它们之间的关系。

## 到底什么是"模态"

模态（Modality）就是信息的呈现形式。大多数人把"多模态"理解为"支持更多输入类型"——似乎只是接口层面的功能扩展。但模态之间的差异远不止格式不同：每种模态承载的信息维度和密度完全不同，它们之间的关系不是"同一种信息的不同包装"，而是"不同维度的信息互相补充"。

| 模态 | 日常例子 | 你以为它在干嘛 | 它实际在干嘛 | 信息密度参考 |
|------|---------|---------------|-------------|-------------|
| 文本 | 文章、聊天、代码 | 传递信息 | 离散符号序列，通过 token 表示语义 | ~0.75 token/字，信息最稠密 |
| 图像 | 照片、截图、海报 | 给人看 | 连续的像素矩阵，空间结构就是信息本身 | 1024×1024 ≈ 256 patch token |
| 音频 | 语音、音乐、环境音 | 让人听 | 时间序列波形，频率+振幅随时间变化 | 10 秒语音 ≈ 数千帧 |
| 视频 | 抖音、电影、监控 | 放片子 | 图像序列 + 时序信息，帧间变化承载运动和叙事 | 1 分钟 30fps ≈ 1800 帧 |
| 3D/传感器 | 点云、雷达、力反馈 | 专业人员才用 | 三维空间中的位置和属性数据 | 取决于采样密度 |

注意表格的"你以为"这一栏——大部分开发者对多模态的理解停留在"把图片转成文字发给 LLM"这个水平。但这恰好是多模态最大的误区：**音频不是"有声音的文字"，图片不是"有像素的文字"**。

一段肖邦的夜曲包含的情感张力、节奏起伏、和声色彩——没有任何文字描述能完全还原。这就是为什么"先转文字再处理"的流水线方案（pipeline approach）在处理真正多模态问题时总会碰壁——你在转换过程中丢弃了原始模态的独特信息。

## 五类产品的多模态能力现状

理解多模态的最好方式，是看产品层面它到底能做什么、不能做什么。2026 年中，主流产品的多模态能力已经高度分化：

### 文生图

最普及的多模态应用——输入文字，输出图像。2026 年已经进入"质量拉不开差距"的阶段，差异在控制力和生态。

| 产品 | 引擎 | 核心能力 | 优势场景 | 定价 |
|------|------|---------|---------|------|
| Midjourney V7 | 自研扩散模型 | 空间理解强、物体关系准确、风格统一 | 概念设计、游戏原画、品牌视觉 | $10-120/月（订阅制，按 GPU 时长计费） |
| DALL-E 3 | OpenAI | 文本跟随性最强、Prompt 理解深 | 产品配图、快速原型、ChatGPT 内集成 | $0.04-0.08/张（Standard），$0.08-0.12/张（HD） |
| gpt-image-2 | OpenAI（GPT-5.5 配套） | LLM 原生集成，上下文感知 | 对话内配图，跨轮编辑 | $0.009-0.17/张（三档质量） |
| Flux Pro 1.1 | Black Forest Labs | 细节真实、文字渲染领先 | 电商图片、广告素材 | $0.04-0.06/张 |
| Stable Diffusion 3.5 | Stability AI（开源） | 可本地部署、LoRA 微调、社区生态 | 定制化需求、私有部署 | 本地免费 + GPU 成本 |

关键差异不在"能不能生成"，而在：DALL-E 3 理解复杂 Prompt 最好，Midjourney 审美和风格最稳定，Flux 写实细节最逼真、文字渲染最强，SD3.5 最灵活可定制，gpt-image-2 在 GPT 生态内上下文理解最深。（数据来源：tokenmix.ai、aicost.ai 2026-06）

### 文生视频

2024-2026 年发展最快的多模态方向。竞争焦点从"能不能动"转向"动作一致性"和"单场景长度"。

| 产品 | 最大单段 | 运动一致性 | 优势场景 | 定价 |
|------|---------|-----------|---------|------|
| Sora 2 (OpenAI) | ~60s | 高，物理规则理解好 | 广告短片、概念展示 | $0.10/s（720p）— $0.70/s（1080p） |
| Seedance 2.0 (字节跳动) | ~15s | 高，多模态 4 输入（文图音视）+ 原生音视频联合生成 | 品牌视频、多镜头叙事、音画同步 | 约 ¥1/s（纯生成）/ ¥0.6/s（编辑模式） |
| Runway Gen-4.5 | ~15s | 中高，Director Mode 跨镜头一致性 | 创意视频、多镜头叙事 | 订阅 $15-95/月 |
| Kling 2.5 (快手) | ~30s | 中，长镜头运动稳定 | 短视频、社交媒体 | 按量计费 |
| Pika 3.0 | ~10s | 中低，风格迁移好 | 风格化短片、动态海报 | $10-58/月 |

Sora 2 在物理世界的理解上领先——它不太会出现"人走路时腿反了"这种低级物理错误。Runway 的 Director Mode 可以保持多镜头中物体和角色的一致性。Seedance 2.0 是唯一支持 4 种输入（文图音视）联合引用的视频模型，独创 @ 引用系统指定每个输入的作用（运动/风格/角色/镜头/音频节奏），还可以原生生成与画面同步的音频（对话、音效、背景音乐）。

但所有视频生成产品在面对手部精细动作、物体交互时仍然不稳定。

### 音乐生成

从"不可能"到"可用的"只用了不到两年。2024 年 Suno V3 让文生音乐进入公众视野，2026 年已经可以生成质量接近商业 Demo 的音乐作品。

**Suno V5.5**：歌词 + 风格描述 → 完整歌曲。支持多段落结构（主歌-副歌-桥段）、多语言歌词、风格指定（"80年代合成器流行"、"J Dilla 风格 Lo-fi Hip Hop"）。2026 年 Suno 付费用户超 200 万，ARR 突破 $3 亿。Pro 计划 $8/月（2500 credits，约 500 首）、Premier $24/月（10000 credits，约 2000 首）。支持 stem 导出（分轨导出人声、鼓点、贝斯等），方便在 DAW 中混音。

**Udio**：与 Suno 功能类似，支持最长 8 分钟，在 EDM/hip-hop 音色质感上略胜。$10/月（Standard）或 $30/月（Pro）。适合快速 ideation，但 stem 导出不如 Suno 灵活。

两家的共同局限：多乐器编配时会"音色混乱"——钢琴和弦乐同时出现时音色互相影响。生成的歌曲无法精细编辑——不能说"第二段副歌加一个鼓点"。（数据来源：suno.com，udio.com 2026-06）

### 多模态对话

这是 2025-2026 年最大的产品进步。不仅仅是"能看图"，而是"图文音频在统一上下文中理解"。

**GPT-5.5（OpenAI）**：文本 + 图像 + 音频（输入）+ 音频（输出 Realtime）+ 视频。2026 年 4 月发布的 GPT-5.5 是首个从零训练的"全模态"（natively omnimodal）基础模型——文本、图像、音频、视频在同一训练中完成，而非后期拼接。单次可处理 1M token 上下文，输入 $5/1M tokens，输出 $30/1M tokens。你可以：拍一张白板照片问"帮我整理成 Markdown"、传一段会议录音问"总结待办事项"、在 Realtime 模式下直接语音对话。视频生成通过 Sora 2 API（$0.10-0.70/s），图像生成通过 gpt-image-2（低 $0.009/张，高 $0.17/张）。

**Gemini 2.5 Pro（Google）**：文本 + 图像 + 音频（原生）+ 视频（原生）+ 1M 上下文，输入 $1.25/1M，输出 $10/1M。原生音频理解意味着不需要"转文字"步骤——模型直接处理音频波形，理解语气、停顿、情感。视频理解支持实时摄像头输入（Live）。最新 Gemini 3.1 Pro 上下文扩展到 2M tokens。

**Claude Opus 4.8（Anthropic）**：文本 + 图像，输入 $5/1M，输出 $25/1M，1M 上下文。2026 年 5 月发布，SWE-bench Pro 69.2%（GPT-5.5 58.6%），编程和深度推理领先。音频支持有限。（数据来源：llm-stats.com、aipricing.guru 2026-06 更新）

### 原生多模态模型

2026 年的新品类：输入和输出都是多模态的，不局限于"理解"。典型代表是 Gemini 的 Imagen 3 集成和 GPT-5.5 的 images 输出模式——同一个模型既能理解图片，也能生成图片，不再需要"LLM + 文生图模型"拼接。

## 单模态模型做不了的事

理解多模态最简单的方法是看单模态模型做不到什么。

**场景一：发票处理。** 你把一张发票照片发给纯文本模型，它不知道你在说什么。发给多模态模型，它能读出发票号码、金额、日期，甚至告诉你"盖章在右下角"。纯文本方案需要：OCR → 结构化解析 → 字段提取。三步串联，每一步都可能出错。多模态方案：一次 API 调用。

**场景二：会议录音处理。** 你录了一段会议录音说"帮我整理待办事项"。纯文本方案：ASR 转文字 → 去口语化 → LLM 提取。但 ASR 会丢失语气信息——那句"这个方案……我觉得可以再考虑一下"中的停顿和犹豫决定了这到底是一个建议还是否决。

**场景三：产品演示视频分析。** 你把一段软件操作录屏发给 AI。纯文本方案：视频怎么转文字？逐帧截图发给多模态模型？但 100 帧的序列需要 100 次 API 调用，而且模型丢失了帧间的时间关系。原生多模态方案：直接把视频作为输入，模型理解"第 3 秒用户点了哪里、第 5 秒发生了什么变化"。

**场景四：多模态输入组合。** 用户发了一张截图 + 一段语音："这个报错是怎么回事？"纯文本方案：必须先把截图 OCR 转文字、语音 ASR 转文字，再拼成一段文字输入。多模态方案：截图直接作为图像输入、语音直接作为音频输入，模型在内部做跨模态关联——它知道截图中被红色框出的区域和语音中"这个"是同一个东西。

## 多模态的技术栈全景

多模态看起来是一个词，但底层的技术栈是三层叠加的。理解这个分层结构，是后续阅读的基础。

**基础能力层（编码与表示）**。各模态的编码/解码。文本怎么切 token、图片怎么切成 patch、音频怎么转频谱图、视频怎么做帧采样。这是所有多模态应用的基础——编码方式决定了 API 账单、信息上限和跨模态处理的难度。对应 06 篇。

**跨模态对齐层（翻译、对齐与融合）**。怎么让不同模态的表示"说得上话"。CLIP 把图文映射到同一向量空间、DALL-E 3 用生成式对齐实现精准文本跟随、注意力机制在 token 级别做跨模态关联。对齐的质量决定了模型"看到的"和"理解的"之间有多大鸿沟。对应 07-09 篇。

**协同学习层（跨模态知识迁移）**。当一个模态数据不足时，怎么从另一个模态借知识。CLIP 的零样本分类——不学任何分类头就能做图像分类；DeViSE 用文本知识改善视觉分类。这个层决定了模型在面对新场景时的泛化能力。对应 10 篇。

**应用能力层**。不同模态组合解决具体问题。文生图（图文→图）、文生视频（文→视频）、文生音乐（文→音频）、Computer Use（截图→动作）、多模态 Agent（多模态输入→多模态输出）。这是后续各章的主题。

## 为什么多模态在 2024-2026 年爆发

多模态的概念在学术界讨论了超过十年，但直到最近两年才真正进入产品化阶段。三个转折点缺一不可：

**转折一：架构的泛化能力。** 2017 年 Transformer 提出时主要针对 NLP。但之后一系列发现证明它是处理多种模态的通用架构底座：ViT（2020）把 Transformer 应用到图像、Whisper（2022）把同样的架构用于音频、VideoMAE（2022）扩展到视频。到 2024 年，纯卷积架构也被证明可以参与多模态处理。**核心结论：你不需要为每种模态设计不同的架构，同一个基础架构经改造后可以覆盖所有模态。** 这大大降低了多模态系统的工程复杂度。

**转折二：数据的规模效应。** CLIP（2021）用 4 亿图文对证明了跨模态对齐可以规模化。再之后 LAION-5B（58 亿图文对）、DataComp-1B（百亿级过滤数据）的出现，让对齐质量持续提升。2025-2026 年，GPT-5.5 和 Gemini 2.5 Pro 的多模态预训练数据达到万亿 token 级别。一个反直觉的事实：**只要数据量够大，模型能自己学会"猫的图片"和"cat 这个词"之间的对应关系，不需要人工标注每个对应关系。** 这让多模态模型从"专家手工设计"变为"数据驱动"。

**转折三：API 的工程化。** 2024 年 GPT-4o 发布，首次把文本 + 图像 + 音频的理解能力集成到单一 API。2025-2026 年，各家跟进。开发者不再需要拼凑 3-5 个不同供应商的服务来做多模态应用。一个 API Key、一次调用，返回统一格式的结果。这让多模态从"能做"变成了"好做"。

三个条件叠加的效果不是线性的——当架构、数据和 API 同时成熟时，产品落地速度是指数级的。2023 年还是"研究论文里的演示"，2024 年是"可用的 Beta"，2025-2026 年已经是"生产级产品"。

## 一个判断标准

不是所有场景都需要多模态。在引入多模态之前，问自己一个问题。一个简单的测试方法：**非文本信息从输入中拿掉，结果还完整吗？**

如果去掉图片只剩文字描述，结果给不出正确判断——你需要多模态。

如果去掉语音只剩文字转录，丢失了关键信息（语气、情感）——你需要多模态。

如果去掉视频只剩文字描述，无法理解操作流程的时序逻辑——你需要多模态。

典型适用场景：
- 输入本身不是文本的（用户的图片、语音消息、视频片段）
- 需要理解信息之间的空间/时间关系（图表布局、视频时序、UI 层级）
- 需要从多种信息通道中交叉验证（合同文字 + 盖章位置 + 手写签名）
- 交互方式越自然越好（语音交互优于打字、截图优于描述）
- 内容是 AI 生成的（你正在用任何文生图/视频/音乐工具）

反之，如果你的应用中所有输入最终都可以无损转化为文字（如纯表格数据、纯 API 返回的 JSON），那么多模态带来的收益可能有限。
