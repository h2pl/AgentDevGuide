# 模态异质性：四种模态的底层结构差异

多模态听起来简单——不就是"支持更多输入类型"吗？你给模型传图片、传音频，它处理，你拿结果。真正做过的人会告诉你不是这么回事。

不同模态的信息天然存在于完全不同的表示空间里，这种差异叫**模态异质性（Modality Heterogeneity）**。这是多模态技术最根本的挑战，不理解它，你就理解不了为什么一张图片能吃掉你 90% 的 token 预算、为什么有些组合成熟有些还在早期、为什么有的模型做图文理解很厉害但完全不能生成音乐。

## 四种模态的底层结构

### 文本：离散的符号系统

文本是人类定义的抽象符号系统。每个"词"是一个离散的 token，通过 tokenizer 切分、embedding 映射成向量。

文本编码的特点：
- **高语义密度**：每个 token 几乎都有明确意义。100 个词的文本 ≈ 100-150 个 token（取决于 tokenizer），覆盖完整句意。
- **低冗余**：没有"废话像素"。如果一段文字压缩到 50 个 token 能表达清楚，它就不会用 100 个 token 来说。
- **表达天花板低**：你没法用文字精确描述一张脸上的微表情、一个声音中的颤抖、一段视频中的气氛变化。

这正是文生图 Prompt 工程困难的原因：文字"夕阳下的沙滩"是一个高度抽象的语义压缩，但 DALL-E 需要把这个语义还原成像素级的视觉呈现。文本的 5 个 token 要在模型中展开成 1024×1024 像素——信息密度差了三个数量级。

**实测参考**：一段 500 字的中文文章 ≈ 380-450 token（GPT-5.5）。一段同样的内容用图像呈现 ≈ 1,000-2,500 token。这就是多模态推理几乎必然比纯文本贵的结构原因——加入任何非文本模态，token 预算都会暴增。

### 图像：空间的像素矩阵

一张 1024×1024 的图片是 1024×1024×3 = 314 万个数字。每个数 0-255，表示颜色强度。

图像和文本的根本区别：**空间结构就是信息本身**。"猫在图片的中央偏左"这种信息不是"描述"出来的，而是直接编码在像素排列中的。你移动一个像素的位置，画面内容可能从"猫"变成"扭曲的色块"——这和文本完全不同（调换两个无关词的顺序可能不影响语义）。

图像的另一个问题：**信息冗余极大**。一张蓝色天空背景 + 一只猫的照片，70% 的像素在描述"不同程度的蓝色"，但模型必须处理所有像素才能找到猫。这直接解释了为什么多模态 API 接收一张图的成本相当于几百个文本 token——模型需要计算的不只是"内容"，还有大量冗余信息。

**图像 token 的量化数据**（以 Qwen2.5 VL 的 tokenizer 为例，2026 年各模型类似）：

| 分辨率 | 约等价 token 数 | 场景参考 |
|--------|---------------|---------|
| 336×336（低成本模式） | 144 | 缩略图、小图 |
| 1024×1024（标准） | 1,369 | 网络图片、截图 |
| 1280×720（高清） | 1,196 | 高清照 |
| 1920×1080（全高清） | 2,769 | 海报、出版级 |
| 3840×2160（4K） | 10,549 | 专业摄影 |

这意味着一张 4K 图像消耗的 token 相当于一份 20,000 字的文档。如果你在应用中对每张用户上传的图片都开启"高分辨率模式"，token 开销会迅速失控。

### 音频：时间的波形

音频不像文本和图像那么直观。一段 16kHz 采样的语音，每秒 16,000 个采样点，每个采样点是对应时刻的声压值。

音频最特别的地方：**信息在时间维度上展开，且多通道并存。** 同一段音频中包含语音内容（说了什么）、说话人身份（谁说的）、情绪状态（语气）、环境信息（在哪里录的）——这些全部编码在同一个波形中。这比图像难处理得多，图像中的人物和背景至少在空间上是可分离的。

文生音乐（Suno/Udio）的技术难点就在这里：生成的音频需要同时保证节奏正确、旋律和谐、音色自然、多乐器协调——所有信息都在同一段波形中存在。

**音频 token 的量化数据**（以 Gemini 的音频处理规格为例）：

| 时长 | 约等价 token 数 | 场景参考 |
|------|---------------|---------|
| 10 秒语音 | 320 | 短语音消息 |
| 30 秒对话 | 960 | 客服录音片段 |
| 1 分钟通话 | 1,920 | 录音片段 |
| 10 分钟会议 | 19,200 | 会议录音 |
| 9.5 小时（上限） | ~1,094,400 | 单次提交上限 |

Gemini 将每秒音频编码为 32 个 token（下采样至 16Kbps，单声道）。OpenAI GPT-Realtime-2 的处理方式不同——流式处理音频帧，音频输入 $32/1M tokens（文本输入 $4/1M），两者相差 8 倍，这正是因为音频 token 的数量远大于文本 token。

### 视频：空间叠时间

视频是最复杂的。30fps、1 分钟 = 1,800 帧图像，加上帧间的时序关系。

Gemini 的实测规格：每帧采样为 258 token，结合 32 token/s 的音频编码，**每秒视频约 300 token**（含元数据）。

| 视频时长 | 约等价 token 数 | 场景参考 |
|---------|---------------|---------|
| 10 秒片段 | ~3,000 | 短视频片段 |
| 1 分钟 | ~18,000 | 产品 Demo |
| 10 分钟 | ~180,000 | 教学视频 |
| 1 小时 | ~1,080,000 | 接近 1M 上下文上限 |

这就是为什么 Gemini 的 2M 上下文窗口对视频理解至关重要——1 小时视频几乎吃掉整个窗口。也是为什么 Sora 生成视频时最困难的是保持帧间一致性——模型不能只看单帧，还要理解第 1 帧和第 60 帧之间"发生了什么变化"。

**注意**：不同模型对视频的采样率不同。Gemini 默认 1 FPS，但你可以设置自定义采样率（如 0.5 FPS 来节省 token）或只分析关键片段。这直接影响成本和质量。

## 异质性带来的三个实战挑战

### 挑战一：维度不匹配

问你一个问题：一张 1024×1024 的图片相当于多少个文本 token？

答案是 **1,000-2,500 个**，取决于模型的分辨率设置。一段 10 秒的音频约 320 个 token（Gemini）。这意味着如果你的 API 调用包含一张图片，图像 token 可能占到全部 token 的 90% 以上。

| 输入内容 | 文本 token | 图像/音频 token | 图像/音频占比 |
|---------|-----------|----------------|-------------|
| 一句话描述 + 一张图 | 50 | 1,369 | 96% |
| 指令 + 一张高清图 + 要求回答 | 150 | 2,769 | 95% |
| 提问 + 10 秒语音 | 80 | 320 | 80% |
| 指令 + 1 分钟视频 | 100 | 18,000 | 99.4% |

模型需要把这些不同维度的输入统一到一个序列中处理。早期融合模型的做法是把图像 patch 和文本 token 拼成一个长序列——这就是为什么多模态调用天然比纯文本贵，这不是定价的问题，是传输和计算的数据量本身就不同。

### 挑战二：信息密度不对等

文本 token 语义密度高但缺感官细节。图像包含丰富的视觉信息但有大量冗余（背景像素）。音频在时间维度上展开，一段 10 秒的语音中可能只有 3 秒在说关键信息。

模型需要识别：
- **哪些是关键的**：图片中的人物、文字、核心物体
- **哪些是冗余的**：纯色背景、重复纹理、无声片段
- **哪些是互补的**：图片中的"红色" + 文字描述中的"紧急"——组合起来才是完整信息

实际影响：当你对一个多模态模型说"这张图片的主体是什么"，模型内部会通过注意力机制聚焦到图像的关键区域（物体所在位置），"忽略"背景区域的向量。但注意力机制本身需要计算，计算量跟序列长度平方相关——所以输入一张大图，即使模型只关注 10% 的像素，它也要先处理 100% 的 token。

### 挑战三：语义鸿沟

"猫"这个文字和一个"有胡须、尖耳朵、毛茸茸的生物"的图片——描述同一个概念，但表示完全不同。文字是抽象符号，图片是具体的视觉特征。

CLIP 训练时做的一件事就是"填平这条鸿沟"——让"猫"的文字向量和猫的图片向量在空间中靠在一起。这条鸿沟填得越好，模型在"我说一个词你找对应的图"这类任务上的表现就越好。

语义鸿沟的不同程度：

| 对齐对 | 鸿沟难度 | 说明 |
|--------|---------|------|
| 文本 → 文本 | 无 | 同一种符号系统 |
| 文本 → 图像 | 中等 | CLIP 已较好解决 |
| 文本 → 语音 | 中等 | Whisper 可处理 |
| 文本 → 音乐 | 高 | 需要抽象描述 → 具体音符和节奏 |
| 文本 → 3D 场景 | 非常高 | 需要从文字重建三维结构 |
| 图像 → 动作指令 | 高 | 需要从视觉推演操作步骤 |

这些鸿沟的深浅直接决定了当前产品成熟度：文生图已非常成熟（因为文字和图像之间的对齐研究最充分），文生视频还在迭代（空间+时间的复杂度），文生 3D 仍处于早期。

## 模态组合成熟度速查

不同模态组合的成熟度差别很大，理解这一点能帮你做选型判断。注意表格中的"实际表现"来自 2026 年 6 月的产品状态，变化很快：

| 组合 | 成熟度 | 代表产品 | 实际表现 | 技术路线 |
|------|--------|---------|----------|---------|
| 文 → 图 | ✅ 非常成熟 | DALL-E 3、Midjourney V7、Flux Pro 1.1、SD3.5 | 日常生产可用，差距在版权和合规 | 扩散模型 |
| 文 → 视频 | ⚠️ 在迭代 | Sora 2、Seedance 2.0、Kling 2.5、Runway Gen-4.5 | 短视频（15-30s）质量好，长视频和时间一致性还在进步。Seedance 2.0 支持 4 种输入联合引用和原生音视频同步 | 扩散模型（DiT）+ 时序建模 |
| 文 → 音乐 | ⚠️ 在迭代 | Suno V5.5、Udio | 短曲目质量惊人，长曲目逻辑和混音还有问题 | 扩散模型 + 音频编解码 |
| 图+文 → 文 | ✅ 非常成熟 | GPT-5.5、Claude Opus 4.8、Gemini 3.1 Pro | 文档分析、视觉问答已上生产。IDP 测评最高 83.5 分 | Transformer (交叉注意力) |
| 语音 → 语音 | ✅ 已可用 | GPT-Realtime-2、Gemini Live | 语音助手可用，延迟 160-320ms。打断还需完善 | 端到端 Transformer + 流式 |
| 图 → 动作 | ⚠️ 可用 | Claude Operator、GPT-5.5 Operator | 截图点击类操作稳定，复杂多步骤流程还不行 | VLM + 动作策略 |
| 视频流理解 | 🔄 早期 | Gemini 3.1 Pro（长视频） | 短视频片段可用（<10 分钟），长时间流推理不稳定 | 帧采样 + Transformer |

**两条核心规律**：

1. **生成侧 vs 理解侧的技术路线完全不同**——生成用扩散模型，理解用 Transformer。GPT-5.5 能做理解和图像生成，但背后是两套完全不同的模型（gpt-5.5 做理解 + gpt-image-2 做生成），通过内部路由打通。Gemini 是唯一做到"同一个模型做理解和生成"的，代价是训练复杂度高得多。

2. **成熟度和异构程度直接相关**——图文对齐（CLIP-style）研究最充分，所以图文理解最成熟。音乐对齐最难做（抽象的"伤感"和具体的音符序列之间鸿沟最大），所以文生音乐虽然可用但远不如文生图稳定。选型时不要用文生图的标准去要求文生视频。
