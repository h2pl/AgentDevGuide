# 视觉 Agent：典型场景与架构模式

视觉能力让 Agent 突破了文本的边界——它现在能"看"屏幕、读文档、识图表。这篇聚焦四个已经产品化的视觉 Agent 场景，给出每个场景的架构模式、模型选型和关键参数。

## 场景一：屏幕 Agent（Computer Use / GUI Agent）

让 Agent 像人一样操作电脑：看屏幕截图、决定点击哪里、输入什么内容。

**架构模式**：

```
截图（800x600，降低分辨率节省 token）
  → 多模态模型分析屏幕（Claude Opus 4.8 或 Qwen2.5-VL）
  → 输出坐标 + 操作（{"action": "click", "x": 450, "y": 320}）
  → 执行操作 → 截图验证结果 → 循环
```

**关键参数**：
- 截图分辨率：800x600 最佳——过高浪费 token，过低看不清细节。Claude 的 grounding 能力在 1024x768 以内表现稳定
- 截图频率：每次操作后截图，每秒不超过 2 次（避免触发反爬/被限流）
- Token 消耗/步：约 2,000-4,000 token（含截图+历史操作上下文）
- 典型延迟/步：Claude Opus 4.8 约 2-3s，Gemini 2.5 Flash 约 0.8-1.2s

**实测表现（2026-06）**：
- 最简单任务（打开浏览器，访问 URL）：Claude Opus 4.8 成功率 95%，平均 3-5 步
- 中等任务（填写表单）：成功率 70-85%，主要失败点在日期选择器和下拉菜单
- 复杂任务（跨系统数据录入）：成功率 40-60%，长链路的累积错误是主要问题

**何时不用**：如果目标系统有 API，优先用 API。Computer Use 是"最后的集成手段"——API 不可用或没有 API 时才上 Computer Use。API 调用的延迟是毫秒级，Computer Use 是秒级。

## 场景二：文档处理 Agent

自动处理 PDF、扫描件、图片中的结构化信息提取。这是目前落地最广的视觉 Agent 场景。

**架构模式**：

```
文档输入
  → 文档分类器（发票/合同/报表/...）→ 每类走不同处理管线
  → 如果是简单表格 → GPT-5-mini 解析（便宜，$0.15/1M input）
  → 如果是复杂图表 → GPT-5.5 或 Gemini 3.1 Pro（准确率优先）
  → 输出结构化 JSON → 验证规则检查 → 人工审核（高价值场景）
```

**文档类型推荐方案**：

| 文档类型 | 推荐模型 | 1 页成本 | 注意事项 |
|---------|---------|---------|---------|
| 发票（标准格式） | GPT-5-mini | ~$0.001 | 准确率约 95%，5% 需要人工修正 |
| 合同（多页长文） | Gemini 3.1 Pro | ~$0.005 | 1M 上下文可处理整份合同 |
| 学术论文（图文混排） | GPT-5.5 | ~$0.003 | 公式理解 Gemini 略强 |
| 手写表单 | Claude Opus 4.8 | ~$0.008 | 手写识别 Claude 明显领先 |
| 财务报表（复杂表格） | Gemini 3.1 Pro | ~$0.004 | 多级表头/合并单元格处理最好 |

**实践要点**：
- 分页处理时保持上下文连续——将前一页的摘要传入下一页
- 对提取结果做规则验证（金额必填、日期格式、枚举值校验）
- 设置置信度阈值，低于阈值自动转入人工审核

## 场景三：视觉 QA / 多模态客服

用户上传图片问问题，Agent 基于图片内容回答。电商售后、医疗影像初筛、工业质检等场景。

**架构模式**：

```
用户上传图片 + 问题
  → 简短上下文缓存（历史对话）
  → 多模态模型推理（GPT-5.5 或 Gemini 2.5 Flash）
  → 如果涉及精确测量/计数 → 调用专用视觉模型验证
  → 返回答案（带引用标注）
```

**Prompt 模板示例**：
```
你是一个[场景]专家。用户上传了一张图片并问了以下问题：
{question}

请基于图片内容回答。如果图片内容不足以回答，请明确指出"从图片中无法确认"。
回答时引用图片中的具体区域（"左下角标签显示..."）。
输出格式：JSON，字段包括 answer、confidence（0-1）、evidence（引用来源）。
```

**关键指标**：
- 首答准确率：目标 > 85%，低于 70% 说明场景不适合视觉模型
- 拒绝率（模型主动说看不清）：5-15% 正常，> 20% 说明图片质量或模型选型有问题
- 人工介入率：控制在 10% 以内

## 场景四：多模态 RAG Agent

在文本 RAG 基础上增加图像索引和检索，支持"找一张去年活动海报"或"这个产品和之前那个有什么不同"这类图文混合查询。

**架构模式**：

```
用户查询
  → 查询分类（图文混合查询 vs 纯文本查询 vs 纯图像查询）
  → 如果是图文混合 → CLIP 双编码器检索（图像 + 文本分别嵌入）
  → 如果是纯图像 → CLIP 图像编码器做相似度检索
  → 检索结果（图文混合）→ 重排序 → 多模态模型合成回答
```

**CLIP 索引配置参考**：
- 嵌入维度：512（ViT-B）或 768（ViT-L），单张图片嵌入约 0.4ms（GPU）
- 索引规模：10 万级用 FAISS（IVF+PQ），百万级用 Milvus/Qdrant
- 混合检索权重：文本相似度 0.6 + 图像相似度 0.4（可调整）
- 多模态模型 Token 预算：检索结果中保留 3-5 张图片 + 对应文字摘要

## 场景选型总结

| 场景 | 推荐模型 | 延迟要求 | 典型 Token/次 | 替代方案 |
|------|---------|---------|-------------|---------|
| 屏幕 Agent | Claude 4.8 / Qwen2.5-VL | <3s/步 | 3,000 | 无替代，必须视觉 |
| 文档处理 | Gemini 3.1 Pro / GPT-5.5 | <5s/页 | 2,000-5,000 | OCR + 文本模型（省 80% 成本） |
| 视觉 QA | GPT-5.5 / Gemini 2.5 Flash | <2s | 1,500 | 专用分类模型（仅分类场景） |
| 多模态 RAG | CLIP + GPT-5.5 | <1s 检索 + <2s 回答 | 2,000 | 纯文本 RAG（无图像则用） |
