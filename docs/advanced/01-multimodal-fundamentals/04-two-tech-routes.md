# 两条技术路线：理解 vs 生成

多模态看起来是一个概念，但底层是两种完全不同的技术路线。**理解侧（Understanding）** 和 **生成侧（Generation）** 从架构到训练到计费都不是一回事。

这个区分对开发者的实际影响比你想象的大：你在用 DALL-E 生成图片时走的是扩散模型链路，在用 ChatGPT 分析图片时走的是 Transformer 链路。它们之间比你想象的独立——独立训练、独立部署、独立计费。即使 GPT-5.5 通过同一套 API 暴露"分析"和"生成"能力，底层调用的仍然是两个完全不同的模型。

## 理解侧：让模型"看懂"和"听懂"

理解侧的任务是：输入图像/音频/视频，输出文字或结构化信息。典型产品：ChatGPT 图片问答、Gemini 视频理解、语音转文字。

**架构：Transformer + 感知编码器**

```
图像 → ViT 编码器 → patch 向量序列 ┐
音频 → 音频编码器 → 帧向量序列    → Transformer（自注意力）→ 文字/结构化输出
文本 → Tokenizer → token 向量序列  ┘
```

核心是你已经熟悉的 Transformer，前面加一个编码器把非文本输入转成向量序列。ViT（Vision Transformer）把图片切成 patch 编码（patch 大小通常 14×14 或 16×16），Whisper 把音频转成序列编码（80 通道梅尔频谱图帧），然后和文本 token 一起过 Transformer。

最新进展是 GPT-5.5 的"原生全模态"（natively omnimodal）架构——所有模态在同一训练中完成，而不是后期拼接。这意味着模型内部的跨模态注意力机制是在训练初始就学到的，而非事后适配。Gemini 3.1 Pro 也采用类似策略。

**推理方式**：一次前向传播。输入送到模型，计算一次，出结果。GPT-5.5 典型延迟 0.5-3 秒（取决于输入大小），Gemini 2.5 Pro 约 1-4 秒。

**计费**：按 token 计费。GPT-5.5 文本输入 $5/1M tokens，图像输入也按 token 计费（一张图 ≈ 200-2,769 token，取决于分辨率），音频按 $32/1M tokens（GPT-Realtime-2）。这就是为什么多模态理解 API 比纯文本贵得多——一张输入图片的"token 成本"可能相当于一整段对话。

**能力边界**：理解侧擅长"识别"和"分析"，不擅长"创造"。Claude Opus 4.8 能准确识别 UI 截图中的每个元素并解释布局逻辑，但让它生成一张一样的 UI——做不了（那是生成侧的事）。GPT-5.5 虽然能同时做理解和图像生成，但走的是两套不同的模型。

## 生成侧：让模型"创造"视觉和听觉内容

生成侧的任务是：输入文字（或图片），输出图像/视频/音乐。典型产品：Midjourney V7、DALL-E 3、Flux Pro 1.1、Sora 2、Suno V5.5、Kling 2.5。

**架构：扩散模型 + 条件引导**

```
文字描述 → 文本编码器（CLIP/T5）→ 条件向量
↓
纯噪声 → 去噪网络（UNet/DiT）← 条件引导（20-50 步迭代去噪）
↓
VAE 解码器 → 清晰图像/视频/音频
```

注意这个过程完全是反直觉的：**从一个纯噪声开始，逐步去掉噪声，最终得到清晰的内容。** 这不是"从无到有"，而是"从乱到治"。每一步都在根据文字条件微调当前输出。

**架构演进：UNet → DiT。** 2023 年之前，扩散模型的主流去噪网络是 UNet（卷积神经网络）。2023 年 Peebles & Xie 提出 Diffusion Transformer（DiT），证明用 Transformer 替换 UNet 可以显著提升生成质量，而且更重要的是——**DiT 的扩展行为是可预测的**（FID 随计算量增加单调下降）。到 2026 年，主流生成模型全面转向 DiT：Sora 2 用 DiT 做视频生成，Flux Pro 用 DiT，Stable Diffusion 3.5 用 DiT，Seedance 2.0 也基于 DiT 架构。DiT 比 UNet 的优势在于：它能更好地处理文本条件、支持更大的模型规模、生成质量随计算量稳定提升。

**推理方式**：20-50 步迭代。每步一次计算，串行执行。典型延迟：DALL-E 3 约 3-10 秒/张，Midjourney V7 约 5-15 秒/张，Flux Pro 约 3-8 秒/张（取决于步数和分辨率）。视频生成延迟长得多：Sora 2 生成 10 秒 720p 视频约 1-3 分钟。

**计费**：各产品差异巨大。按张数（DALL-E $0.04-0.12/张，gpt-image-2 $0.009-0.17/张），按月订阅（Midjourney $10-120/月），按秒（Sora 2 $0.10-0.70/秒）。和 token 消耗完全无关——这也是理解侧和生成侧在成本模型上的根本区别。

**能力边界**：生成侧擅长"创造"，不擅长"分析"。Flux 能生成一张"戴帽子的猫坐在打字机前"的图片，但让它识别图片里有什么——做不了。

## 为什么这个区分很重要

你可能会想：GPT-5.5 不是既能分析图片又能生成图片吗？它应该是统一的吧？

不是。GPT-5.5 的"统一"是在 API 层面的——你通过同一个 API 传入"分析"和"生成"请求，但底层调用的是两个完全不同的模型：

```
你: "分析这张图有什么问题"
    ↓
GPT-5.5 API → 路由到 GPT-5.5（Transformer 理解模型）→ 分析结果

你: "生成一张夕阳下的海滩图片"
    ↓
GPT-5.5 API → 路由到 gpt-image-2（扩散模型）→ 图片
```

开发者通过 `tool_choice: { type: "image_generation" }` 触发图像生成，背后是路由系统在决定调用哪个模型。理解侧用 GPT-5.5（Transformer），生成侧用 gpt-image-2（扩散模型）。两者独立训练、独立部署、独立更新、独立计费——甚至不在同一个数据中心集群上。

**只有 Gemini 是例外。** Google Gemini 系列是唯一真正实现了"同一个模型做理解和生成"的路线——Gemini 3.1 Pro 可以接受图片、分析内容，也可以直接生成图片（通过 Imagen 3 集成）。但这种统一是有代价的：训练复杂度更高、模态间的权衡更难做（优化文本理解可能损害图像生成质量）。GPT-5.5 的"路由"方案更实际，但代价是你不能在一个模型中同时获得理解侧和生成侧的推理能力。

### 对选型的实际影响

**文生图/视频/音乐场景，只看生成模型。** Midjourney V7 的审美一致性、Flux Pro 的英文字母渲染精度、Sora 2 的物理世界理解——这些是各自扩散模型训练出来的，和理解侧模型在 Arena 视觉榜上的排名没有关系。

**分析/理解场景，只看理解模型。** Claude Opus 4.8 在 SWE-bench Pro 上的优势（69.2%）、GPT-5.5 在 Terminal-Bench 2.0 上的表现（82.7%）、Gemini 3.1 Pro 的超长 2M 上下文——这些都是 Transformer 架构的能力。扩散模型完全无法参与这些评测。

**两套预算体系。** 理解侧按 token 优化（降分辨率、裁剪图片、限音频时长、用 Batch API 打 5 折）。生成侧按步数优化（用蒸馏模型减少采样步数、降低输出分辨率）。优化策略完全不同，意味着做多模态应用的团队可能需要两个专业的工程师分别管两条链路。

### 一个例外：端到端语音模型

GPT-Realtime-2 是最近打破"理解 vs 生成"二分法的例子。它直接处理音频帧 → 输出音频帧，在一个模型中完成语音识别 + 语言理解 + 语音合成。它不是简单的"理解 + 生成"拼接，而是端到端训练的音频原生模型。类似的产品还有 Gemini Live。但对开发者来说，Realtime API 仍然走独立的计费链路（$32-64/1M tokens），和文本级 API 不互通。

## 生成侧的三个核心挑战

### 控制精度

"一只戴帽子的橘猫"——指令很简单，但扩散模型怎么确保生成的内容精确匹配？条件引导的质量决定了控制精度。ControlNet、LoRA 等技术在 2025-2026 年大幅提升了控制能力，让你可以精确定位画面中每个元素的位置。

### 速度

扩散模型需要 20-50 步迭代才能从噪声还原出图像。每一步是一轮 UNet/DiT 计算。速度优化从三个方向突破：
- **少跑几步**：采样算法优化（DPM++、LCM），把需要的步数从 50 降到 3-5
- **每步跑快**：模型蒸馏、量化，减少每步的计算量
- **并行处理**：视频生成时可以并行处理多个帧，但帧间一致性做不好

### 视频生成的时间一致性

这是 2026 年最活跃的研究方向之一。单帧生成已经很好，但相邻帧之间的物体一致性（人不能突然消失、背景不能突变）是视频生成的核心难点。Sora 用时空 patch 把"帧"的概念泛化了，其他方案（Kling、Runway）则是在扩散过程的每一步加入时序约束。

## 两条路线的全面对比（2026-06）

| 维度 | 理解侧 | 生成侧 |
|------|--------|--------|
| 核心架构 | Transformer（自注意力） | 扩散模型（迭代去噪），现正转向 DiT |
| 典型代表 | GPT-5.5、Claude Opus 4.8、Gemini 3.1 Pro | DALL-E 3、Flux Pro 1.1、Sora 2、Suno V5.5 |
| 信息流 | 图文/音频输入 → 文字/结构化输出 | 文字输入 → 图像/视频/音频输出 |
| 推理方式 | 一次前向传播 | 20-50 步迭代（图），50-100+ 步（视频） |
| 典型延迟 | GPT-5.5: 0.5-3s，Gemini 3.1 Pro: 1-4s | 3-10s（图），1-3 min（视频 10s） |
| 计费方式 | 按 token（GPT-5.5: $5/$30 per 1M） | 按张/秒（DALL-E: $0.04-0.12/张，Sora: $0.10-0.70/s） |
| 优化方向 | 降 token 消耗（降分辨率、裁剪、限音频） | 降采样步数（蒸馏、LCM、DPM++），减分辨率 |
| 训练方式 | 自回归掩码语言建模 + 对齐训练 | 前向加噪 + 反向去噪 + 条件引导 |
| 当前瓶颈 | 跨模态对齐精度（尤其长视频和复杂推理） | 生成速度 + 控制精度（尤其视频帧一致性） |
