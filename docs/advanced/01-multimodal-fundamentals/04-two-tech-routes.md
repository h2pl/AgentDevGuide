# 两条技术路线：理解 vs 生成

前几篇讲了多模态的认知框架，但有一个根本性的区分必须先搞清楚——**多模态 AI 不是一套统一技术，而是两条完全不同的技术路线拼在一起的。**

**理解侧（Understanding）** 让模型"看懂"和"听懂"：输入图像/音频，输出文字或结构化信息。**生成侧（Generation）** 让模型"创造"：输入文字描述，输出图像/视频。它们使用的架构、训练方式、推理过程截然不同。

理解这一点很重要——在实际开发中，理解任务和生成任务几乎总是：不同的 API、不同的计费方式、不同的模型权重、不同的延迟表现。把"多模态"当成一套统一技术来理解，会在选型和预算上产生严重误判。

## 理解侧：Transformer + 感知编码器

理解侧的代表是 GPT-5.5、Claude Opus 4.8、Gemini 2.5 Pro 这些模型。它们的本质是**大语言模型加了一个"感知前端"**：

```
图像/音频/视频 → 感知编码器（ViT / Audio Encoder）→ 向量序列
↓
文字输入 → Tokenizer → 向量序列 → LLM（Transformer）
↓
文字/结构化输出
```

核心架构和你已经熟悉的 LLM 是同一个 Transformer。只不过在纯文本 Transformer 前面加了一个视觉编码器（ViT）或音频编码器，让模型能"看到"和"听到"。

理解侧的推理过程是**一次前向传播**：模型对输入进行一次计算，输出结果。这个过程很快，典型延迟在 0.5-3 秒。

理解侧有三个核心机制：**编码**（把图片/音频变成向量）、**融合**（让不同模态的向量交互）、**对齐**（匹配跨模态的对应关系）。接下来的三篇文章（06-08）会逐篇深入这三个机制。

## 生成侧：扩散模型 + 条件引导

生成侧的代表是 DALL-E 3、Stable Diffusion 3.5、Flux、Sora。它们用的架构完全不同——**扩散模型（Diffusion Model）**：

```
文字描述 → 文本编码器（CLIP / T5）→ 条件向量
↓
纯噪声 → 去噪网络（UNet / DiT）← 条件引导（逐步去噪 20-50 步）
↓
VAE 解码器 → 清晰图像/视频
```

核心是**迭代去噪**过程：从一个纯噪声图像开始，按照文字条件的引导，每步去掉一点噪声，逐步还原出清晰的图像。这个过程需要 20-50 步迭代，所以生成侧的延迟远高于理解侧。

生成侧的三个核心挑战：
- **怎么精确控制生成内容**。条件引导的质量决定了"一只戴帽子的橘猫"能不能真的生成一只戴帽子的橘猫
- **怎么提高质量和速度**。采样算法优化、模型蒸馏、并行计算——每项技术都在推低生成侧延迟
- **怎么扩展到视频**。视频生成需要保持帧间一致性，这比单张图像生成难一个数量级

这些会在[图像与视频生成](../04-multimodal-generation/01-image-and-video-generation.md)中系统讲解。

## 两条路线的全面对比

| 维度 | 理解侧 | 生成侧 |
|------|--------|--------|
| 核心架构 | Transformer（自注意力） | 扩散模型（迭代去噪） |
| 信息流方向 | 多模态输入 → 文字输出 | 文字输入 → 图像/视频输出 |
| 推理方式 | 一次前向传播 | 20-50 步迭代去噪 |
| 典型延迟 | 0.5-3 秒 | 3-30 秒（图片）/ 1-5 分钟（视频） |
| 计费方式 | 按 token 计费 | 按张数/时长计费 |
| 优化方向 | 降低 token 消耗 | 减少采样步数 |
| 当前瓶颈 | 跨模态对齐质量 | 生成速度 + 控制精度 |

## "理解"和"生成"能整合到一个模型吗

2026 年有一个明显的趋势：API 层面正在走向统一。Gemini 通过 Imagen 3 把生成能力集成到了同一个 API，GPT-5.5 也能在一次对话中混合调用理解和生成。你通过同一个接口传入"帮我分析这张图"的请求和"帮我生成一张图"的请求。

但底层仍然是两套引擎。同一个 API 背后，理解任务走 Transformer 链路，生成任务走扩散模型链路。它们之间不共享模型权重，只是 API 调度层做了统一封装。

这对你意味着什么：
- **接口简化了**。不需要维护两套 API 的接入代码
- **选型逻辑没变**。理解任务看 Arena 视觉榜选模型，生成任务看图像质量选模型
- **成本结构不同**。理解按 token 计费，生成按张数计费，优化方向完全不同
