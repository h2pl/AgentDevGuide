你好，我是江小湖。上一篇讲了多模态的全景——它是什么、为什么需要、现在哪些模型支持。你可能好奇：模型内部到底怎么处理这些完全不同类型的数据？一张图片和一段文字，怎么就能在同一个模型里"互相理解"？

这篇你进入模型的内部。但在讲具体机制之前，有一个根本性的区分必须先搞清楚——多模态 AI 其实是两条完全不同的技术路线拼在一起的。

## 两条技术路线：理解与生成

多模态 AI 的能力可以分为两大类：**理解**和**生成**。它们听起来像是一件事的两面，但实际上背后的技术架构、训练方式、推理过程完全不同。

### 理解侧：Transformer + 感知编码器

理解侧的代表是 GPT-5、Claude、Gemini 这些模型。它们的本质是**大语言模型加了一个"感知前端"**：

```
图像/音频/视频 → 感知编码器（ViT / Audio Encoder）→ 向量序列
                                                         ↓
文字输入 → Tokenizer → 向量序列 → LLM（Transformer）→ 文字/结构化输出
```

核心架构是 **Transformer**，和你已经熟悉的 LLM 是同一个东西。只不过在纯文本 Transformer 前面加了一个视觉编码器（比如 Vision Transformer）或音频编码器，让模型能"看到"和"听到"。信息流方向是：多模态输入 → 编码 → 推理 → 文字输出。

理解侧的三个核心挑战是：**编码**（怎么把图片/音频变成向量）、**融合**（怎么让不同模态的向量交互）、**对齐**（怎么匹配跨模态的对应关系）。这些是本文要重点讲的。

### 生成侧：扩散模型 + 条件引导

生成侧的代表是 DALL-E 3、Stable Diffusion、Flux、Sora。它们用的是完全不同的架构——**扩散模型（Diffusion Model）**：

```
文字描述 → 文本编码器（CLIP / T5）→ 条件向量
                                      ↓
纯噪声 → 去噪网络（UNet / DiT）← 条件引导（逐步去噪 20-50 步）
                                      ↓
                               VAE 解码器 → 清晰图像/视频
```

核心架构是**去噪扩散过程**，和 Transformer 完全不同。信息流方向是：文字描述 → 条件引导 → 从噪声逐步还原 → 图像/视频输出。

生成侧的核心挑战是：**怎么精确控制生成内容**（条件引导）、**怎么提高质量和速度**（采样算法、模型蒸馏）、**怎么扩展到视频**（时间一致性）。这些会在[图像与视频生成](../04-multimodal-generation/01-image-and-video-generation.md)系统讲解。

### 两条路线的关键区别

| 维度 | 理解侧 | 生成侧 |
|------|--------|--------|
| 核心架构 | Transformer（自注意力） | 扩散模型（迭代去噪） |
| 信息流 | 多模态输入 → 文字输出 | 文字输入 → 图像/视频输出 |
| 推理方式 | 一次前向传播（快） | 20-50 步迭代去噪（慢） |
| 典型模型 | GPT-5、Claude、Gemini | DALL-E 3、Flux、Sora |
| 典型延迟 | 0.5-3 秒 | 3-30 秒（图片）/ 1-5 分钟（视频） |
| 训练数据 | 图文配对 + 指令微调 | 大规模图像/视频 + 文字描述 |
| 开发者接口 | Chat Completion API | Image Generation API |

**为什么要区分这两条路线？** 因为你在实际使用时，它们几乎总是分开的 API、分开的计费、分开的模型权重。即使 GPT-5 能同时做理解和生成，底层也是两套引擎在跑。把"多模态"当成一套统一技术来理解，会产生很多误判——比如以为理解了编码融合对齐就能搞定生成，其实完全不是那么回事。

不过 2026 年有一个趋势：Gemini 通过 Imagen 3 把生成能力集成到了同一个 API 接口，GPT-5 也能在一次对话中混合调用理解和生成。API 层面正在走向统一，但底层引擎仍然是两套。

本文接下来聚焦**理解侧**的核心机制。生成侧的系统讲解在[图像与视频生成](../04-multimodal-generation/01-image-and-video-generation.md)。

## 理解侧三阶段：编码→融合→对齐

以下三个机制是理解侧（Transformer 体系）的核心。CMU 多模态课程把多模态信息处理分成三个阶段：

```
输入（图片/文字/语音）→ 编码 → 融合 → 输出（回答/动作/生成）
```

- **编码**：每种模态各自变成向量表示
- **融合**：不同模态的向量在同一个空间里交互
- **对齐**：确保不同模态的信息能正确对应

这三个阶段不是三个独立步骤——在现代原生多模态模型里，它们往往是交织在一起的。但分开理解每个机制，能帮你看清全貌。

## 编码：把不同模态变成模型能理解的向量

编码（Representation / Encoding）是多模态处理的第一步：**把每种模态的原始数据转换成模型能计算的数字向量。**

### 文本编码

这个你在[第 2 章 LLM 基础](../../02-llm-basics/01-llm-overview.md)里已经见过了。文本通过 Tokenizer 切成 token 序列，每个 token 对应一个嵌入向量（embedding）。比如"猫"这个字可能变成 `[0.23, -0.15, 0.87, ...]` 这样的 4096 维向量。

### 图像编码

图像编码的方式和文本完全不同。主流方法是**把图片切成小块（patch）**，每个小块单独编码成一个向量：

- 一张 224×224 的图片，切成 14×14 = 196 个 patch
- 每个 patch 经过 Vision Transformer（ViT）变成一个向量
- 最终得到 196 个向量，每个描述图片的一个局部区域

这就是为什么图像会消耗大量 token——一张图可能就占了几百个 token 的预算。后面讲[视觉理解](../02-multimodal-vision/01-vision-understanding.md)时会细讲成本优化。

### 音频编码

音频编码又有不同的方式：

- **语音**通常先转换成梅尔频谱图（Mel Spectrogram）——本质上是把声音的时频特征可视化，然后用类似图像处理的方式编码
- **Whisper 类模型**直接把音频波形编码成向量序列，和文本 token 放在同一个空间里处理

### 为什么编码方式很重要

不同模态的编码方式决定了它们的信息密度和计算成本：

- 文本编码最"紧凑"——一个 token 对应一个有意义的词或子词
- 图像编码最"冗余"——196 个 patch 向量描述一张图，但很多 patch 可能只是背景
- 音频编码最"连续"——声音信息在时间上展开，需要处理长序列

当你对比不同模型的延迟和成本时，底层原因往往就藏在编码方式里。

## 融合：把不同模态的信息合到一起

编码之后，每种模态都有了自己的向量表示。但这些向量还在各自的空间里，"猫的图像向量"和"cat 的文字向量"之间没有直接联系。

融合（Fusion）就是把这些不同来源的信息合到一起，让模型能做跨模态的理解。

### 三种融合策略

**早期融合（Early Fusion）**：在最底层就把不同模态的向量拼在一起，让模型从头开始联合处理。

```
[图像 patch 向量] + [文本 token 向量] → 拼接成一个长序列 → 一起过 Transformer
```

这是 GPT-4o、Gemini 这类原生多模态模型的做法。优势是模态之间从最底层就能交互，模型能编码"这个图片区域对应这段文字描述"这种细粒度关系。代价是训练成本高，需要大量多模态配对数据。

**晚期融合（Late Fusion）**：每种模态各自独立处理到最后一步，只把最终结果合在一起做判断。

```
图像 → 视觉编码器 → 图像特征向量 ┐
                                    → 拼接 → 分类/回答
文本 → 语言编码器 → 文本特征向量 ┘
```

这是流水线方案的做法。优势是简单、灵活、可以复用单模态模型。劣势是模态之间的交互太晚，无法利用细粒度的跨模态关系。

**交叉注意力（Cross-Attention）**：一种折中方案——每种模态保持自己的处理路径，但在关键层通过注意力机制"看"另一种模态的信息。

```
文本 Transformer 层 ←→ Cross-Attention ←→ 图像 Transformer 层
```

CLIP、Flamingo、LLaVA 等模型用了这种方式。文本模型和视觉模型各自独立编码，但在某些层通过交叉注意力让文本 token 可以"关注"到相关的图像 patch。

### 怎么判断你用的模型是哪种融合方式

- **原生多模态模型**（GPT-5、Gemini 2.5）：早期融合，所有模态从底层就在一起处理
- **视觉语言模型**（LLaVA、Qwen-VL）：交叉注意力，视觉编码器独立，通过投影层接入语言模型
- **流水线方案**：晚期融合，各模态独立处理后拼接结果

融合方式直接影响模型的跨模态推理能力。早期融合的模型在需要深度跨模态理解的任务上（比如"图片里的猫在做什么，和这段文字描述一致吗"）表现最好。

## 对齐：匹配跨模态的对应关系

对齐（Alignment）是多模态中最精妙的挑战：**怎么让模型正确关联"这张猫的图"和"cat 这个词"说的是同一个东西？**

### 对比学习：CLIP 的思路

OpenAI 的 CLIP（2021）是对齐领域的里程碑。它的训练方式出奇地简单：

1. 收集大量"图片+文字描述"配对数据（比如互联网上的图文对）
2. 同时训练一个图像编码器和一个文本编码器
3. 目标：让配对的"图片向量"和"文字向量"在向量空间中靠近，不配对的互相远离

训练完成后，CLIP 的图像编码器和文本编码器产生**同一个向量空间中的表示**——"猫的图片"和"cat 这个词"的向量余弦相似度很高，和"狗"的向量相似度很低。

### 为什么对齐重要

对齐质量直接决定了多模态模型的能力上限：

- **对齐好**：模型能准确理解"图片里有一只橘猫坐在沙发上"，因为"橘猫"的文字向量和图片中猫的区域向量高度对应
- **对齐差**：模型可能把图片里的猫识别成狗，因为它没有建立起足够精确的跨模态对应关系

### 预训练对齐 vs 微调对齐

- **预训练对齐**：在大规模数据上学习通用的跨模态对应关系（CLIP、SigLIP）
- **指令微调对齐**：在高质量的"指令+多模态输入+回答"数据上微调，让模型能按用户指令做跨模态推理（LLaVA、GPT-4V 的训练方式之一）

大多数现成模型已经帮你做好了这两步对齐。但如果你在用开源模型做微调，对齐质量就是最需要关注的指标——对齐没做好，模型在多模态任务上的表现会比纯文本差很多。

## 从分治到统一：原生 vs 拼接的本质区别

现在你可以把编码、融合、对齐三个机制串起来，理解"原生多模态"和"拼接式方案"的本质区别了：

| 维度 | 拼接式方案 | 原生多模态 |
|------|-----------|-----------|
| 编码 | 每种模态用不同模型独立编码 | 统一编码器或联合编码 |
| 融合 | 晚期融合：各自编码后拼接结果 | 早期融合或交叉注意力 |
| 对齐 | 依赖外部对齐（如 CLIP）或不做对齐 | 训练过程中学习对齐 |
| 跨模态推理 | 弱：各模态信息在文本层面拼接 | 强：模态间从底层就交互 |
| 典型代表 | OCR + Whisper + Vision + LLM | GPT-5、Gemini 2.5 Pro |

简单说：拼接方案是"先各自理解，再把结论合起来"；原生方案是"从一开始就一起理解"。后者能做到前者做不到的细粒度跨模态推理，但也需要更多的训练数据和算力。

## 理解机制对开发者有什么用

你可能会问：我又不训练模型，知道这些有什么用？

用处比你想象的大：

**调试幻觉**：模型在图片理解上出现幻觉，很多时候是因为图像编码的 patch 粒度不够细（比如小字看不清），或者对齐没做好（模型把图里的 A 对象和 B 描述搞混了）。知道这些机制，你能更快定位问题。

**成本优化**：图像 token 贵是因为编码后产生大量 patch 向量。如果你知道这个原理，就能理解为什么缩小图片分辨率、裁剪无关区域能降低成本。

**选型决策**：需要深度跨模态推理的场景（比如医疗影像分析），优先选早期融合的原生模型。只需要浅层理解的场景（比如提取图片中的文字），拼接方案可能更经济。

**理解局限**：知道对齐的原理，你就明白为什么模型有时候会"指鹿为马"——它在向量空间中把鹿的特征和"马"这个概念靠得太近了。这不是 bug，是对齐过程中不可避免的近似。

## 总结

多模态 AI 不是一套统一技术，而是两条不同的技术路线：

- **理解侧**（Transformer 体系）：编码 + 融合 + 对齐，核心是把多模态输入变成文字输出
- **生成侧**（扩散模型体系）：条件引导 + 迭代去噪 + VAE 解码，核心是把文字描述变成图像/视频

本文详细讲了理解侧的三个核心机制：

- **编码**：把文本/图像/音频各自变成向量，不同模态的编码方式决定了信息密度和计算成本
- **融合**：把不同模态的向量合到一起，从晚期融合到早期融合，跨模态交互的深度逐级增加
- **对齐**：建立跨模态的对应关系，让模型知道"这张图"和"这个词"说的是同一个东西

生成侧的系统讲解在[图像与视频生成](../04-multimodal-generation/01-image-and-video-generation.md)，那里会用同样的深度拆解扩散模型的完整机制。
