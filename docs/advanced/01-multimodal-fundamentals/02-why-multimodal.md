# 为什么需要多模态：流水线方案的三个结构性缺陷

单模态方案在 2024 年之前是唯一选择——那时候没有成熟的多模态 API，你只能把图片转成文字喂给 LLM，把语音转成文字再处理。但 2026 年的今天，继续用这套流水线方案（pipeline approach）的性价比已经很低了。

先看几个真实场景，从最常见的开始。

## 场景一：文生图——你已经离不开多模态

你用 Midjourney V7 生成海报（$10-120/月）、用 DALL-E 3 做配图（$0.04-0.12/张）、用 Flux Pro 1.1 生成电商素材（$0.04-0.06/张）。这是最典型的多模态应用：输入文字，输出图像。2026 年的图像质量已经接近真假难辨——Midjourney V7 的人物面部一致性大幅改善，Flux 的英文字母渲染准确率超过 95%，SD3.5 的 LoRA 社区生态超过 10 万个微调模型。

这个场景本身就是多模态的证明——你不需要把"需求"先转成什么中介格式，模型直接从文字跳到图像。但这里有一个隐蔽的坑：**文生图模型和 LLM 是两个完全不同的技术栈。** ChatGPT/Claude 用 Transformer + 自回归解码（预测下一个 token），底层芯片以 GPU 计算为主。Midjourney/SD 用扩散模型（从噪声逐步去噪到图像），需要不同的 CUDA kernel 和显存模式。计费方式也完全不同——LLM 按 token 计费，文生图按张数或 GPU 时长计费。这就是为什么 GPT-5.5 的 gpt-image-2 是一个独立的 API，而不是 ChatGPT 对话的一个隐藏输出模式——底层架构无法合并。

## 场景二：分析一份带图表的 PDF 合同

你让 Agent 分析一份合同。合同里有文字条款、图表数据、手写签章、公司 logo。如果你的 Agent 只能处理文本，流水线方案长这样：

```
PDF → OCR 提取文字 → 视觉模型描述图表 → 签名验证
                                              ↓
                                     LLM 综合 → 回答
```

每个环节跑在不同的 API 上、各自计费、各自有延迟、各自会出错。我们用 2026 年的数据和标准来量化问题有多严重。

### 问题一：信息逐级丢失是结构性的

每一环的转换都在丢失信息。OCR 从图片中提取文字时，字体颜色、排版位置、手写批注的语义、盖章的大小和位置——这些要么变成文字被简化掉，要么完全丢弃。

等所有中间结果拼到一起交给 LLM 时，原始图片中的"这一页的右下角有一个红色的圆形印章"已经被简化成"有印章"三个字（如果 OCR 识别到"印章"这个词的话）。而"印章的位置是否覆盖了签名区域"这种问题，流水线方案天生答不了——它没有保留空间信息。

**数据支撑：** IDP Leaderboard（智能文档处理排行榜，2026-06）对主流多模态模型的文档理解能力做了全面评测。GPT-5.4 综合得分 83.5，Gemini 3 Pro 82.8，Claude Sonnet 4.6 80.7。但这是原生多模态模型的分数——如果走 OCR+LLM 流水线，在同样的评测集上，最优流水线方案得分约低 15-20 个百分点，因为 OCR 环节对表格结构、手写文字、盖章区域的处理误差会逐级放大。

### 问题二：错误会叠加

OCR 把"张三"识别成"张二"——这个错误会传递到最终回答，而且难以定位。多模型拼接的调试复杂度是 O(n)——每个环节都可能出错，你需要在 n 个日志中追溯。

按 2026 年的数据做一个量化估算：主流 OCR API（Google Document AI、Azure Form Recognizer）的中文识别准确率约 97-99%。但 PDF 合同中通常包含：正文文字（5,000 字）、表格数据（3,000 字）、手写签名（1 处）、盖章文字（2-3 处）。按 98% 逐字准确率计算，5,000 字正文约 100 个错字，加上 OCR 对手写和盖章区域几乎必定出错。这些错误全部累积后输入 LLM，LLM 还要在此基础上做推理，结果可靠性可想而知。

### 问题三：上下文断裂

模型看不到"原始文档"什么样。它只能看到拆解后的文字碎片。"合同的第二页第三行的金额条款是什么"——流水线方案需要额外开发位置追踪机制，而原生多模态模型在内部就解决了这个问题。

## 原生多模态方案的做法

```
PDF 合同 → 多模态模型（一次 API 调用）→ 文字内容 + 图表数据 + 签章位置 → 回答
```

核心区别不是"少调几个 API"，而是模型内部的跨模态注意力机制让文字 token 可以直接"关注"到图片中对位的 patch 向量。模型能看清"第三页右下角的签章处有盖章，红色，圆形"——因为它确实在内部同时看到了视觉信息与文本信息。

GPT-5.5 在 IDP 评测中综合得分 83.5，处理一份 20 页 PDF 的延迟约 3-5 秒，成本约 $0.05-0.15（按输入 50K tokens 计算）。流水线方案：OCR（$0.01-0.03，1-3 秒）+ 图表视觉模型（$0.01-0.02，1-2 秒）+ LLM 综合（$0.005-0.01，0.5-1 秒），总成本 $0.03-0.06 且总延迟 2.5-6 秒，看似便宜但质量明显更低。多模态模型的额外成本换来的是信息完整度——你花 $0.03 得到的是 OCR 筛选过的文字摘要，花 $0.05 得到的是完整的文档理解。（数据来源：openai.com/api/pricing、idp-leaderboard.org 2026-06）

## 场景三：语音交互的故事

你做一个语音助手。流水线方案：

```
用户语音 → ASR（Whisper）→ 文字 → LLM 处理 → 文字 → TTS（语音合成）→ 语音回复
```

三个核心问题在 2026 年仍然存在：

1. **串行延迟**：ASR + LLM + TTS 三段累加。Whisper 约 300-500ms（5 秒语音），GPT-5.5 首 token 约 200-500ms，TTS 约 300-500ms。合计 **800ms-1.5s**，勉强可接受但无法自然打断。Realtime API 端到端延迟为 160-320ms。

2. **情绪信息丢失**：用户用焦虑的语气说"这个方案真的可行吗"，ASR 提取的是中性文字"这个方案真的可行吗"，LLM 无法感知焦虑。Realtime API 直接处理音频帧，保留语调、语速、停顿信息。

3. **打断难实现**：用户中途想补充，流水线方案必须等当前 TTS 输出完才能处理新输入。Realtime API 支持流式中断——当你说话时，模型会暂停输出并处理新输入。

**成本对比**：2026-06 数据，GPT-Realtime-2 的音频输入 $32/1M tokens，音频输出 $64/1M tokens。一段 30 秒的对话（输入约 15 秒音频，输出约 20 秒音频）约 $0.008-0.012。流水线方案（Whisper $0.006/分钟 + GPT-5.5 $5-30/1M + TTS $0.015/分钟）总成本约 $0.003-0.005/次，但功能上少了情绪感知和自然打断。多模态方案的成本约 2-3 倍，换来的是更像人的交互体验。（数据来源：openai.com/api/pricing、tokennix.ai/blog/openai-realtime-voice-2026）

## 场景四：文生视频和音乐

Sora 2、Seedance 2.0、Kling 2.5、Runway Gen-4.5 这类文生视频产品，以及 Suno V5.5、Udio 这类文生音乐产品，天然就是多模态的——输入文字，输出视频或音乐。它们没有一个"流水线方案"的替代选项。你不能用 OCR + LLM + ASR 拼出一个文生视频功能。这就是多模态的另一个面向：**有些能力只有原生多模态才能做**。

理解侧的流水线方案只能说"质量差一点但勉强能用"，视觉生成侧的扩散模型和音频生成模型没有替代方案。Suno V5.5 能生成带多段落结构（主歌-副歌-桥段）的完整歌曲，Sora 2 能生成 60 秒连贯视频，Seedance 2.0 支持 4 种输入联合引用和原生音视频同步——这些能力的底层全部依赖原生多模态架构。

2026 年的行业格局已经清晰：**文生图/视频/音乐已经不存在"要不要用多模态"的选择问题，而是"用哪个产品"的选择问题。** 具体比较见[01 篇](./01-what-is-multimodal.md)各产品表格。

## 三组数字帮你决策

不是所有场景都需要原生多模态。以下三组对比帮你判断，数据均来自 2026 年 6 月各家官网定价和第三方评测：

| 场景 | 流水线方案 | 原生多模态方案 | 差距 |
|------|-----------|---------------|------|
| 分析带图 PDF | OCR + 视觉模型 + LLM，3 次 API，0.03-0.06 美元/次，延迟 2.5-6s | 1 次 API 调用，0.05-0.15 美元/次，延迟 3-5s | 成本 2-3x，但 IDP 评分高 15-20 个百分点 |
| 语音助手 | Whisper + GPT + TTS，0.003-0.005 美元/次，延迟 0.8-1.5s | GPT-Realtime-2，0.008-0.012 美元/次，延迟 0.16-0.32s | 成本 2-3x，但支持情绪感知和打断 |
| 文生图/视频/音乐 | ❌ 没有替代方案 | 原生支持 | 没有可比性 |
| 视频分析（1 分钟） | 逐帧 OCR + 帧串联 + LLM，120+ 帧 x 多次 API，延迟 > 30s | Gemini 原生视频，1 次 API，延迟 5-10s | 延迟降低 70-80%，成本降低 50%+ |

结论：流水线方案在 2024 年之前是必要的妥协。2026 年它还在两个场景下有意义：
1. **极端预算限制**——流水线用最便宜的模型拼起来可能比原生多模态 API 便宜（如语音场景成本约 1/3）
2. **需要控制单个环节**——合规要求必须使用特定 OCR 或 ASR 服务（如金融行业必须使用认证的 OCR 方案）

其他场景，原生多模态方案在质量、延迟、维护成本三个维度上都更优。
