# 为什么需要多模态

先看一个真实场景。

你让 Agent 帮你分析一份 PDF 合同。合同里有文字条款，也有图表数据、手写签章、公司 logo。如果你的 Agent 只能处理文本，它需要走这样一条流水线：

```
PDF → OCR 提取所有可见文字 → 视觉模型描述图表内容
                                  ↓
签名验证模型检查盖章位置 → LLM 综合所有文本结果 → 回答
```

每一步是一个独立的 API 调用，数据在服务之间通过文本传递。

## 流水线方案的三个核心问题

### 问题一：信息逐级丢失

每一轮转换都在丢失信息。OCR 从图片中提取文字时，手写批注的语义、签章的位置、图表的空间结构、文字的字体和颜色——这些信息要么被丢弃，要么被简化成一段文字描述。等到所有中间结果拼到一起交给 LLM 时，原始文档中可能超过一半的信息已经没了。

"信息丢失"不是模糊的概念。一张表格的"数字"和"项目名称"在空间上相邻，这种空间关系经过 OCR 后完全消失。LLM 只能根据拼接后的文字猜测"这个 15000 到底对应哪个项目"。

### 问题二：管线脆弱

五个不同的服务串行工作。任何一个环节出错都会污染最终结果：

- OCR 把"张三"识别成"张二" → 后续所有判断基于错误数据
- 视觉模型没识别出图表类型 → 图表数据完全丢失
- LLM 综合时上下文长度超限 → 部分信息被截断

更棘手的是错误定位。用户拿到一个错误答案，你很难判断是 OCR 错了、视觉模型漏了、还是 LLM 推理错了。每个服务都可能出错，但你没有单次 API 调用的端到端一致性保证。

### 问题三：上下文断裂

模型看不到"原始文档"是什么样的——它只能看到被拆解后的文字碎片。文字和图表的对应关系、签章在文档中的位置、文字之间的排版逻辑，这些信息在拆解过程中全部丢失了。

这个问题在需要"定位"的场景下尤其致命。比如用户问"合同第三页的签名处有没有盖章"，流水线方案需要额外写代码来追踪页码和位置信息，而多模态模型直接"看"一眼就知道了。

## 原生多模态方案的做法

多模态模型的处理方式完全不同：

```
PDF 合同图片 → 多模态模型（一次 API 调用）
                    ↓
              文字内容 + 图表数据 + 签章位置
                    ↓
              综合分析回答
```

不需要拆成多个步骤。模型在内部做跨模态注意力计算时，文字 token 可以直接"关注"到图片中对应位置的 patch 向量。这意味着模型能说清"第三页右下角的签章处有盖章"——它确实"看到"了那个位置。

这不是"把 OCR 和 LLM 合并到一个 API"那么简单。两者的本质区别在于：

| | 流水线方案 | 原生多模态 |
|--|-----------|-----------|
| 信息传递方式 | 服务间通过文本传递 | 模型内部跨模态注意力 |
| 错误率叠加 | 每个服务独立出错，累积 | 单次推理，端到端优化 |
| 空间关系保留 | ❌ 转换为文本后丢失 | ✅ 保持像素级对应 |
| 延迟 | 串行调用 3-5 次 API | 一次 API 调用 |
| 维护成本 | 每个服务独立升级、计费 | 单一 API，统一计费 |

## 什么时候可以不用多模态

有些场景不需要多模态。判断方法很直接：**输入去掉非文本信息后仍然完整，就只需要文本模型。** 如果去掉后丢失了关键信息，多模态就是必要的。

典型的"不需要"场景：

- 输入已经是结构化数据（JSON、数据库记录、CSV）
- 任务只需要文本能力（翻译、摘要、纯代码生成）
- 延迟要求极高，连一次多模态 API 的延迟都接受不了
- 预算极有限，图像 token 成本确实比文本 token 高

## 实践决策的思考方式

2026 年做技术选型时，"要不要用多模态"的决策应该反过来想：不是"有没有必要用多模态"，而是"不用多模态的话，我需要额外写多少代码来处理非文本信息"。

原本需要拼凑 OCR + ASR + 视觉模型 + 签名验证四个服务的场景，现在一次多模态 API 调用就能覆盖——除非你有极端特殊的需求（比如必须纯本地部署且多模态模型不支持你的场景），否则原生方案在质量、延迟、维护成本三个维度上都是更好的选择。
