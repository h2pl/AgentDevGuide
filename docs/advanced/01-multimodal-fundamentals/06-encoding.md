# 多模态编码

多模态处理的第一步，是把每种模态的原始数据转换成模型能计算的向量。这个过程叫做**编码（Encoding）**。

一篇讲编码，不需要把它讲得很复杂。你只需要理解三件事：

1. 不同模态的编码方式完全不同
2. 编码方式直接决定了成本和性能
3. 知道这些能在实际使用中帮你省钱、定位问题

## 三种编码方式

### 文本编码：最直接的方式

文本通过 Tokenizer 切成 token，每个 token 查表得到一个嵌入向量：

```
"猫在睡觉" → Tokenizer → ["猫", "在", "睡觉"] → embedding lookup → 3 个向量
```

这是三种编码中最成熟的。每个 token 承载一个明确的语义单位，信息密度很高。100 个词的文本 ≈ 100 个 token，token 消耗和语义量大致成正比。

### 图像编码：切 patch

图像编码的方式完全不同。主流做法是用 **Vision Transformer（ViT）** 把图片切成小块（patch），每个小块单独编码：

```
224×224 图片 → 14×14 = 196 个 patch → 每个 patch 编码成一个向量 → 196 个向量
```

这个"切 patch"的操作是理解图像 token 消耗的关键。一张 224×224 的中等分辨率图片产生 196 个向量。如果分辨率提高到 1024×1024，patch 数量翻 20 倍以上，token 消耗也同比上涨。

图像编码的两个特点直接影响到你的实际使用：

**空间结构保留。** 每个 patch 带有位置信息，模型知道"左上角的 patch 对应图片左上角的内容"。这是多模态模型能说清"桌子的左边是一把椅子"这种空间关系的原因。

**信息冗余。** 196 个 patch 中很多只是背景。但模型必须处理所有 patch 才能理解图片——它不知道哪些是"重要的"哪些是"背景"，直到做完推理。

### 音频编码：转频谱图

音频编码常见的做法是把波形转成**梅尔频谱图（Mel Spectrogram）**——本质上是一张"时间 × 频率"的图像，然后用 ViT 或 CNN 编码：

```
10 秒音频 → 分帧 → FFT → 梅尔频谱图 → 编码器 → 向量序列
```

另一种方式（Whisper 类模型）直接把音频波形编码成向量序列，和文本 token 放在同一个空间里处理。

音频编码的核心挑战是**序列长度**。10 秒的语音在频谱图中产生数千个时间帧，远超过同样时长文本的 token 数。这就是为什么音频处理的 token 消耗比文本高得多。

## 编码方式决定了三件事

### 成本

这是你最先感受到的差异。文本 token 最便宜、消耗最少。图像 token 贵——一张 1024×1024 的图片 ≈ 几千个文本 token。音频按时长计费，10 秒语音的 token 消耗大约是 100 个文本 token 的量级。

知道了这个，你就理解为什么缩分辨率、裁剪区域是最有效的省钱手段——不是模型在处理上更高效，而是输入给模型的 patch 减少了。

### 信息上限

编码方式决定了模型能从输入中"看到"多少信息。小分辨率图片的 patch 粒度不够细，模型看不清小字。低采样率音频丢失高频信息，模型听不清某些辅音。

如果你发现模型总在某个细节上出错，可以检查一下输入分辨率/采样率是否足够。

### 跨模态处理的难度

编码后的不同模态向量在维度、分布上都不相同。文本向量来自 embedding table，图像向量来自 ViT，需要后续的融合机制来统一处理——这就是下一篇的内容。

## 实践建议

| 操作 | 效果 | 适用场景 |
|------|------|----------|
| 缩小图片分辨率 | patch 数减少，token 降低 50-90% | 不需要精细细节的场景 |
| 裁剪无关区域 | 保留关键区域分辨率，消除冗余 | 图片中有大量背景的场景 |
| 限制音频时长 | 按需截取关键片段 | 长录音中只有前 15 秒相关的场景 |
| 降低图像颜色深度 | 部分模型支持灰度图输入，减少处理量 | OCR、文档扫描等颜色无关场景 |
