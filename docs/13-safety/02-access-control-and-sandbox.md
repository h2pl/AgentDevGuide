# 访问控制与沙箱执行

> Agent 的权限管理不是"要不要给"，而是"最小给多少"。沙箱执行确保即使 Agent 被攻破，伤害范围也控制在最小。

你好，我是江小湖。上一篇文章解决了"攻击者能不能骗 Agent"的问题。但即使 Agent 被攻破了，**一个设计良好的权限系统可以让攻击者做不了什么**。

就像办公楼的安全：门禁卡能进大厅，但不能进 CEO 办公室；能进自己工位，但不能进服务器机房。Agent 的权限管理也应该遵循同样的原则。

## Agent 权限模型

Agent 的权限模型与传统权限管理（RBAC/ABAC）有共通之处，但也有独特的挑战。

### 传统权限 vs Agent 权限

| 维度 | 传统系统 | Agent 系统 |
|------|---------|-----------|
| 执行主体 | 固定用户/角色 | LLM 动态决策 |
| 权限判断时机 | 静态配置 | 运行时动态判断 |
| 权限粒度 | URL/方法级别 | 工具/参数/数据级别 |
| 风险特征 | 可预测 | 不可预测（LLM 行为不确定性） |
| 审计难度 | 低 | 高（自然语言决策难以自动化审计） |

### 核心原则

**最小权限 (Least Privilege)**：Agent 默认没有任何权限，只为明确需要的能力授予权限。

**纵深防御 (Defense in Depth)**：权限不是"一层墙"，而是多层防线。每层独立失效才能被突破。

**可审计 (Auditable)**：每一次权限使用都记录在案，事后可以追溯。

## 权限分层设计

  <img src="../../assets/13-safety/permission-layers.svg" alt="权限层次模型" width="95%"/>
</p>

Agent 的权限应该分为三个层次，**从底层到顶层逐级收窄**：

```
操作级权限（最严格）
  └── 删除、修改配置、转账等危险操作
      └── 需要人工审批

数据级权限（中等）
  └── 读/写特定数据范围
      └── 按用户维度、数据敏感度分级
      
工具级权限（基础）
  └── 能调用哪些工具
      └── 每个工具独立授权
```

## 工具级权限

工具级权限是 Agent 权限管理的基础——**Agent 只能调用明确授权的工具**。

### 工具清单管理

每个 Agent 上线前，必须维护一个显式的工具清单：

```yaml
agent: customer-service
allowed_tools:
  - query_order        # 查询订单
  - check_shipping     # 查物流
  - update_address     # 修改收货地址
  - cancel_order       # 取消订单（需要审批）
  
restricted_tools:
  - delete_user        # ❌ 不可用
  - query_all_users    # ❌ 不可用
  - execute_sql        # ❌ 不可用
  - send_http_request  # ❌ 不可用
```

**工具清单应该是白名单模式**——不在列表中的工具不可用。黑名单模式（列出不可用的工具）不可靠，因为你会漏掉新加的工具。

### 参数级约束

即使工具被授权了，参数也可能需要限制：

```json
{
  "tool": "query_order",
  "allowed_params": {
    "order_id": "*",            // 任何订单 ID 都可以查
    "user_id": "{caller_id}"    // 只能查自己的用户 ID
  }
}
```

参数约束可以用模板变量（如 `{caller_id}`）实现动态权限——`query_order` 工具可以查任何订单，但「用户只能查自己的订单」这个约束由系统保证，而不是让 Agent 自觉遵守。

### 调用频率限制

为每个工具设置调用频率限制，防止异常行为：

| 工具 | 每用户/分钟 | 每用户/小时 | 全局/分钟 |
|------|-----------|-----------|---------|
| query_order | 30 | 200 | 1000 |
| cancel_order | 2 | 10 | 50 |
| update_address | 5 | 30 | 200 |

## 数据级权限

Agent 能「看到」什么数据，决定了它的知识边界。

### 用户维度隔离

**最基本的数据隔离规则：Agent A 的数据对 Agent B 不可见。** 除非显式共享。

用户隔离的实现：

- **多租户**：每个用户的向量数据库索引分离
- **会话前缀**：对话历史按用户 ID 分片
- **检索过滤**：检索时自动附加 `user_id` 过滤条件

### 数据敏感度分级

将数据分为三个敏感度等级：

| 级别 | 定义 | 示例 | 访问要求 |
|------|------|------|---------|
| L1 - 公开 | 无敏感信息 | 产品目录、常见问题 | 无限制 |
| L2 - 内部 | 一般敏感 | 订单记录、物流信息 | 用户身份认证 |
| L3 - 敏感 | 高度敏感 | 支付信息、身份证号 | 特定 Agent + 人工审批 |

**Agent 在被授权访问 L3 数据时，应该自动触发审计日志和审批流程**。

### 数据脱敏

Agent 在返回敏感数据前，系统层自动脱敏：

```
用户问："我的手机号是多少？"

Agent 内部看到："13800138000"
Agent 回答中呈现："138****8000"
```

脱敏应该在 Agent 输出之后、返回用户之前，由系统层强制执行——**不依赖 Agent 自觉脱敏**。

## 操作级权限

操作级权限控制 Agent "能做什么事"——不仅仅是调什么工具，而是操作的性质。

### 操作分级

| 操作等级 | 定义 | 示例 | 控制方式 |
|---------|------|------|---------|
| 读取 | 无副作用 | 查天气、搜文档、查订单 | 自动执行 |
| 写入 | 有副作用但可逆 | 保存设置、更新草稿 | 自动执行 + 审计 |
| 变更 | 有副作用不可逆 | 取消订单、删除文件 | 人工审批 |
| 高危 | 高影响操作 | 转账、删除用户、修改权限 | 人工审批 + 双人确认 |

### 风险评分

对每个操作计算风险评分，根据评分动态决定是否需要审批：

```
风险评分 = 
  操作影响(1-10) × 
  数据敏感度(1-3) × 
  操作量级(1-5)
  
评分 > 50 → 需要人工审批
评分 > 100 → 需要双人审批 + 72 小时冷静期
```

### Human-in-the-Loop (HITL) 审批

对于变更和高危操作，**Agent 不能独立完成，必须等待人工确认**。

```
Agent: "用户申请取消订单 ORD-123"
系统: "取消订单 ORD-123 需要审批，已通知人工"
人工审批: "同意"
系统: "Agent 执行取消操作"
```

HITL 的关键实现细节：

- **超时机制**：人工审批超过 N 分钟未响应，Agent 自动回复用户"正在等待处理"
- **审批通道**：通过独立通道通知人工（如 IM 消息、邮件），不在 Agent 对话中内置审批
- **审计追踪**：审批人、审批时间、审批结果全部记录
- **紧急回滚**：审批通过的操作如果错误，要有回滚机制

## 沙箱执行

当 Agent 需要执行代码或访问外部资源时，沙箱是最后一道防线。

### 什么是沙箱

沙箱是一个**受限的执行环境**，即使 Agent 在沙箱中做了任何操作，也不会影响外部系统。

类似于 Docker 的安全原理：容器内的 root 不等于主机的 root。

### 沙箱级别

| 沙箱类型 | 隔离强度 | 性能开销 | 适用场景 |
|---------|---------|---------|---------|
| 进程级 | 低 | 低 | 脚本执行、数据处理 |
| 容器级 (Docker) | 中 | 中 | 代码执行、API 调用 |
| 虚拟机级 (Firecracker) | 高 | 高 | 不可信代码、第三方插件 |

**Agent 生产的代码执行，推荐使用容器级沙箱**——隔离强度足够，性能损耗可控。

## 代码执行沙箱

当 Agent 有写代码能力（如 Code Interpreter、Code Agent），代码执行必须在沙箱中。

### 安全策略

```yaml
sandbox:
  network: isolated          # 默认无网络
  filesystem: ephemeral      # 文件系统临时，退出清空
  cpu_limit: 1               # 最多 1 核
  memory_limit: 512MB        # 最多 512M 内存
  timeout: 60s               # 最多运行 60 秒
  allowed_imports:           # Python 包白名单
    - pandas
    - numpy
    - json
    - math
    - re
  blocked_imports:           # 强制禁用包
    - os
    - subprocess
    - socket
    - requests
```

**关键规则**：白名单和黑名单并用。白名单决定能做什么，黑名单兜底阻止已知危险包。

### 无状态沙箱

每个沙箱实例应该是**无状态的**——每次代码执行后自动销毁，不保留任何状态。持久化数据通过显式的输入输出接口传递：

```
用户: "分析一下这份 CSV"
→ 文件上传到沙箱的 /input/ 目录
→ Agent 代码在沙箱中运行，读取 /input/data.csv
→ 结果写入 /output/result.json
→ 沙箱销毁
→ /output/result.json 被提取返回给用户
```

这样即使沙箱被攻破，攻击者也无法持久化访问。

## 文件系统隔离

文件系统隔离比网络隔离更容易被忽视，但同样重要。

### Agent 文件访问模型

```
Agent 工作目录: /tmp/agent-data/{session_id}/  （临时，会话结束清空）
  ├── input/      输入文件（只读）
  ├── output/     输出文件（写后读取）
  └── workspace/  工作目录（读写，临时）

系统文件: 不可访问
其他用户的文件: 不可访问
```

### 文件操作限制

| 操作 | 允许范围 | 说明 |
|------|---------|------|
| 读取 | workspace/ 和 input/ | Agent 只能读自己的文件 |
| 写入 | workspace/ 和 output/ | 只能写自己的目录 |
| 执行 | workspace/ | 只能执行自己的脚本 |
| 网络读取 | 禁止 | Agent 不能 curl/wget 外部文件 |

## 网络隔离

### 默认策略

```
出站: 禁止（默认）
入站: 禁止（默认）
白名单: 根据需求开放特定域名/IP
```

### 域名白名单示例

```
允许: api.weather.com
允许: api.youcompany.com
允许: search.yourcompany.com

禁止: *
```

## 权限审计

所有权限操作都应该有日志记录。

### 审计日志结构

```json
{
  "timestamp": "2026-06-18T10:00:00Z",
  "trace_id": "req_abc123",
  "user_id": "user_456",
  "agent_id": "customer-service-v3",
  "action": "call_tool",
  "tool": "cancel_order",
  "params": {"order_id": "ORD-123"},
  "decision": "need_approval",
  "approver": "admin_789",
  "approval_time": "2026-06-18T10:05:00Z",
  "result": "approved"
}
```

### 审计重点

- **所有变更和高危操作**的完整审计
- **权限变更日志**（谁在什么时候给什么 Agent 加了什么权限）
- **异常模式**检测：同一用户在短时间内大量调用敏感工具

> **审计日志不可篡改**。推荐使用 append-only 存储，或者发送到独立的日志系统（如 SIEM），Agent 自身无权访问和修改审计日志。

## 总结

权限控制和沙箱执行是 Agent 安全的两道实体防线。**权限控制限制"能做什么"，沙箱限制"能做到什么程度"**。

权限分层（工具→数据→操作）确保每个层级独立管控；沙箱（代码/文件/网络）确保即使前两层都被突破，破坏范围仍然可控。
