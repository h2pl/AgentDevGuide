# 输出过滤与人工审批

> Agent 的输入可以过滤，权限可以控制，但最终用户看到的是 Agent 的输出——这里出问题，前面的防御就白做了。输出过滤 + 人工审批是最后两道门。

你好，我是江小湖。前两篇文章建立了一个逐步收紧的防御体系：先防止注入，再限制权限，最后用沙箱兜底。但即使这些都做到了，**Agent 的输出仍然可能出问题**——幻觉、敏感内容、错误的决策。

输出过滤和人工审批是 Agent 安全的两道"最后防线"。

## 为什么输出过滤很重要

输出过滤不是"可有可无"的安全加分项，而是生产系统必须有的基础设施：

**防止敏感信息外泄**。Agent 在回复中可能无意透露了其他用户的信息、内部 API 详情、模型本身的 system prompt——这些应该在到达用户之前被拦截。

**防止有害内容**。即使 Agent 本意是好的，LLM 的输出仍然可能包含不合适的内容——偏见、歧视、不当建议。输出过滤是"说了不该说的"这道防线的最后机会。

**维护用户信任**。用户可能对 Agent 的回答有错误判断。一个自信满满的错误回答（幻觉）对用户信任的损害，比一个"我不知道"的回答严重得多。

## 输出过滤的类型

### 敏感信息过滤

检测和屏蔽输出中的敏感数据：

```json
{
  "sensitive_patterns": [
    {"type": "api_key", "pattern": "[A-Za-z0-9]{32,}"},
    {"type": "email", "pattern": "[\\w.]+@[\\w.]+"},
    {"type": "phone", "pattern": "1[3-9]\\d{9}"},
    {"type": "id_card", "pattern": "\\d{18}[\\dXx]"}
  ]
}
```

**关键设计**：过滤应该是"替换"而不是"阻断"。直接阻断会让用户困惑。把敏感内容替换为脱敏版本：

```
原始: "您的手机号是 13800138000"
过滤后: "您的手机号是 138****8000"
```

### 有害内容过滤

对输出进行内容安全检测：

- **仇恨言论**：基于种族、性别、宗教的歧视性内容
- **暴力内容**：鼓励或美化暴力行为
- **不当建议**：违法的、不道德的建议
- **诱导提示**：输出中包含诱导用户做不当行为的提示

### 事实性检查

对 Agent 输出中的关键事实做自动化验证。最简单的做法：**把 Agent 的回答发回给 LLM 做快速验证**。

```
输出: "根据我们的记录，您的订单将在 2026 年 6 月 20 日送达"

验证: 
  验证器检查 → 订单 ORD-123 的预计送达时间确实是 2026-06-20
  验证结果: ✅ 通过
```

对于可以查证的事实（订单状态、时间、价格），**总是做自动化验证**。对于无法查证的（分析性回答），标记为「未验证」。

## 过滤实现策略

### 过滤层级

输出过滤应该分层实现，每层关注不同的问题：

```
Layer 1: 正则/模式匹配
  ├── 手机号、邮箱、身份证、API Key
  ├── 速度: 微秒级
  └── 误报: 可能有

Layer 2: 内容分类器
  ├── 仇恨言论、暴力、不当内容
  ├── 速度: 毫秒级
  └── 可用小型分类模型

Layer 3: LLM 验证
  ├── 事实性检查、安全合规
  ├── 速度: 秒级
  └── 只在必要时启用
```

**分层策略**：L1 和 L2 对所有输出启用，L3 只对高风险操作启用。

### 过滤时机

过滤应该在输出返回到用户之前，**在 Agent 的执行管道中**完成：

```
Agent 生成回复
  → L1 过滤（正则）
    → 命中？→ 脱敏后继续
  → L2 过滤（分类器）
    → 命中？→ 阻断 + 记录审计
  → L3 检查（高危操作才启用）
    → 命中？→ 触发人工审批
  → 输出给用户
```

### 过滤结果的处理

| 过滤结果 | 动作 | 用户看到的 |
|---------|------|-----------|
| 安全 | 正常输出 | Agent 的回答 |
| 敏感内容 | 自动脱敏 | 脱敏后的回答 |
| 有害内容 | 阻断输出 | "抱歉，我无法回答这个问题" |
| 高风险 | 触发审批 | "这个操作需要等待人工确认" |

## 幻觉检测

幻觉是 Agent 输出中最难检测的问题——因为内容看起来"合理"但不正确。

### 检测方法

**置信度评分**。让 LLM 对自己的回答做置信度评估：

```
用户: "公司去年的营收是多少？"
Agent: "根据 2025 年年报，公司营收为 12.8 亿元"
置信度: 85%（有数据支撑）

用户: "公司明年的营收预测？"
Agent: "预计将达到 15 亿元"
置信度: 30%（推测性内容）
```

置信度低于阈值的回答，可以追加免责声明或触发人工审核。

**引用验证**。对引用来源的回答，验证引用是否真实存在：

```
Agent: "根据我们的文档（docs/internal.md），API 调用限制为 1000次/分钟"
验证: 检查 docs/internal.md 中是否真的有这句话
结果: ✅ 引用存在，内容一致
```

**检索增强生成 (RAG) 自查**。Agent 返回答案后，用答案作为 query 再检索一次，看检索结果是否支持答案内容。如果不支持，答案可能是幻觉。

## Human-in-the-Loop 审批机制

部分操作不应该由 Agent 独立决策。**Human-in-the-Loop (HITL)** 是 Agent 安全体系的最后保险。

### 何时需要 HITL

- **高危工具调用**：删除用户、批量操作、发送通知
- **高影响决策**：退款、赔偿、账户封禁
- **不可逆操作**：删除数据、修改权限
- **低置信度高风险**：Agent 对自己的回答没信心，但操作影响大
- **新场景**：Agent 遇到了从未见过的用户请求

### HITL 的模式

```
模式 1: 先审后执 (Pre-Approval)
  Agent 生成操作请求 → 人工审批 → Agent 执行
  
模式 2: 先执后审 (Post-Approval)
  Agent 执行 → 人工审核结果 → 确认或回滚
  
模式 3: 监督模式 (Supervised)
  Agent 执行 → 人工实时监督 → 随时介入
```

模式 1 最安全但延迟最高，模式 2 最灵活但风险最高。**推荐高影响操作用模式 1，低影响但不可逆操作用模式 2**。

### 审批通知

审批请求通过独立渠道发送给人工审批者：

```
审批渠道（推荐优先级）:
1. IM 消息（钉钉/飞书/企业微信）— 最快响应
2. 邮件 — 中等响应
3. 内部审批系统 — 正式记录
```

审批通知应包含：

- **上下文**：用户请求了什么，Agent 建议怎么处理
- **风险说明**：操作的风险等级、涉及的数据、可能的后果
- **决策选项**：同意 / 拒绝 / 修改后执行
- **超时时间**： 如果 N 分钟内不处理，默认是拒绝还是挂起

### 审批超时

```
审批超时策略:
  ├── 普通操作: 5 分钟内无响应 → 拒绝操作
  ├── 紧急操作: 1 分钟内无响应 → 转第二审批人
  └── 高风险操作: 永不自动批准，必须人工决定
```

### 审批日志

每个审批操作都应该记录完整信息：

```json
{
  "approval_id": "appr_789",
  "trace_id": "req_abc123",
  "user": "user_456",
  "agent": "customer-service-v3",
  "requested_action": "cancel_order ORD-123",
  "risk_score": 65,
  "approver": "admin_789",
  "decision": "approved",
  "approved_at": "2026-06-18T10:05:00Z",
  "note": "用户身份验证通过，确认是本人操作"
}
```

## 紧急回滚

如果 Agent 执行了错误操作，回滚机制是最后的补救措施。

### 可回滚操作设计

**在设计工具时，就应该考虑回滚**：

```
# 不好的设计：直接删除
delete_user(user_id) → 不可回滚

# 好的设计：软删除 + 延时清除
deactivate_user(user_id) → 可恢复
  └── 7 天后自动完全清除 → 可取消
```

每个工具都应该有对应的「撤销操作」：

| 操作 | 撤销操作 | 撤销窗口 |
|------|---------|---------|
| 发送邮件 | 无法撤销 | - |
| 更新订单 | 版本回退 | 24 小时 |
| 删除用户 | 软删除恢复 | 7 天 |
| 修改配置 | 配置历史回退 | 30 天 |

### 回滚触发

触发方式：审批者在审批后 N 分钟内可以一键回滚。

```
Agent已执行: 取消订单 ORD-123
人工检查发现: 取消错了（用户A实际上是用户B的订单）
→ 审批者点击"回滚"
→ 系统自动调用恢复订单接口
→ 订单 ORD-123 恢复为"已取消"之前的"已支付"状态
```

## 安全事件的响应流程

当安全事件发生时（检测到攻击、发现敏感信息泄露），应该有标准的响应流程。

### 响应步骤

```
1. 检测 → 安全告警触发
2. 评估 → 影响范围？涉及哪些用户？什么数据？
3. 阻断 → 暂停 Agent 服务或限制权限
4. 取证 → 导出相关审计日志和追踪数据
5. 修复 → 修复漏洞、更新防御策略
6. 复盘 → 改进防御体系
7. 通知 → 必要时通知受影响的用户
```

### 告警分级

| 级别 | 定义 | 响应时间 | 响应方式 |
|------|------|---------|---------|
| P0 | 敏感数据泄露 / 系统被攻破 | 立即 | 暂停服务 + 全组响应 |
| P1 | 成功注入攻击 / 越权操作 | 15 分钟 | 隔离受影响的 Agent |
| P2 | 注入检测到但被拦截 | 4 小时 | 评估是否需要调整防御 |
| P3 | 误报 / 无实质风险 | 24 小时 | 记录并在下次迭代修复 |

## 总结

输出过滤和人工审批是 Agent 安全体系的最后两道防线。**输出过滤拦截"不该说的"**——敏感信息、有害内容、幻觉。**人工审批拦截"不该做的"**——危险操作、不可逆变更。

结合前面两篇文章的 prompt 注入防御和权限控制，一个完整的 Agent 安全体系应该是：

```
┌─ 输入层: Prompt 注入检测 + 指令隔离
├─ 执行层: 权限控制 + 沙箱
└─ 输出层: 输出过滤 + 人工审批 + 回滚
```
