你好，我是江小湖。前两篇文章建立了追踪系统。但追踪不是目的——目的是**让系统更快、更便宜、更可靠**。

本文讲一件事情：怎么用 Trace 数据找到性能瓶颈和成本黑洞，然后系统性地优化。

核心思路是三步：**测量 → 识别 → 优化**。先量化现状，再找最大改进点，最后针对性优化。不要跳步。

## 性能分析：从 Trace 到瓶颈

### 标准分析流程

当用户反馈"Agent 有点慢"时，按这个流程定位：

**第一步：确认是普遍问题还是个别问题**

看 P50 / P95 / P99 延迟：

```
如果 P50 正常、P99 很高 → 个别慢请求的问题（优化尾部延迟）
如果 P50 也高了 → 普遍变慢（找到所有请求的共同原因）
```

**第二步：找到最慢的 Span**

对一批慢请求，按 Span 类型聚合耗时：

```
查询: 过去 1 小时内 P95 > 5s 的请求
按 Span type 聚合总耗时占比:

llm_call:       68% ← 最大瓶颈
tool_call:      18%
retrieval:      10%
framework:       4%
```

**第三步：下钻到具体的慢 Span**

找出占大头的那类 Span，再看具体是哪个操作慢：

```
llm_call 中:
  gpt-4o:  平均 2.1s  ← 占 LLM 时间大头
  gpt-4o-mini: 平均 0.8s
  o4-mini: 平均 3.2s

tool_call 中:
  query_order:  平均 450ms
  send_email:   平均 120ms
  external_api: 平均 1.8s  ← 这里慢了
```

**第四步：优化影响最大的项**

不要优化占比小的。把精力花在影响最大的瓶颈上。

### Trace 能直接告诉你的事

有了结构化的 Trace 数据，很多问题不需要"猜测"，可以直接回答：

```
"过去一周 P95 延迟升了 40%，是不是换模型了？"
  → 查 Trace 数据，对比两周的 llm_call 耗时分布
  → 发现 gpt-4o-mini 占比从 60% 降到了 30%
  → 根因：模型路由规则被意外修改

"成本翻倍了，是流量涨了还是每请求贵了？"
  → 查 Trace 数据
  → 请求量只涨了 20%，但每请求 token 涨了 60%
  → 根因：有 30% 的请求把 20 轮对话历史全塞进上下文
```

## Agent 特有的延迟模型

Agent 系统的延迟和传统 API 完全不同——它不是一次请求-响应，而是**多轮循环**。理解这个延迟模型是性能分析的基础。

### 延迟构成

```
一个 Agent 请求的总延迟 =
  多次 LLM 调用的串行延迟
+ 每次 LLM 调用的等待时间（TTFT + 生成时间）
+ 工具调用的网络延迟
+ 检索延迟
+ 框架开销
```

关键在于**串行**——Agent 通常是一轮一轮执行的，下一轮依赖上一轮的输出。5 轮循环，每轮 2s，总延迟就是 10s。这不是并行能解决的。

### 五种常见延迟模式

**模式一：LLM 首 Token 等待（TTFT 高）**

现象：LLM Span 很长，但绝大部分时间是 TTFT。

根因：模型太大、prompt 太长、并发太高导致排队。

缓解：换更快的模型、减少输入 token、水平扩展。

**模式二：上下文窗口膨胀**

现象：越往后 Step，LLM 的输入 token 越来越多，耗时越来越长。

```
Step 1: 1200 tokens → 0.8s
Step 2: 2400 tokens → 1.5s
Step 3: 4800 tokens → 2.8s  ← 越来越慢
Step 4: 7200 tokens → 4.2s
```

根因：每轮把全部对话历史塞进 prompt，没有做压缩或裁剪。

缓解：对话历史压缩、窗口滑动、摘要替换原始历史。

**模式三：工具调用链过深**

现象：Agent 为了完成一个简单任务，调用了 5-6 个工具。

根因：Agent 的决策策略有问题，把单步能完成的拆成了多步。

缓解：重构工具粒度、在 prompt 中引导"尽量一步完成"、限制最大工具调用次数。

**模式四：死循环**

现象：同一种工具被反复调用，每次参数略有不同。

根因：Agent 在某个状态卡住了，既没有成功也没有失败，也不知道什么时候该停。

缓解：设置最大步数硬上限、检测重复模式自动终止。

**模式五：外部依赖慢**

现象：工具调用的 Span 中，大部分时间花在等待外部 API 返回。

根因：外部服务慢或不可靠。

缓解：加缓存、设超时、考虑异步。

### 延迟排查清单

当用户反馈"Agent 慢"时，按这个清单快速定位：

```
□ 整体 P95/P99 是否显著变化？
□ 哪类 Span 占总延迟比例最高？
□ LLM 的 TTFT 是否异常高？
□ 输入 token 是否每轮都在膨胀？
□ 工具调用链是否过深？
□ 是否有工具在反复调用？
□ 外部 API 是否有超时或重试？
```

## 分层优化策略

找到瓶颈后，按层优化。每层的优化手段不同。

### 模型层

```
优化手段                      影响
────────────────────────────────────────
模型路由（小模型处理简单请求）   成本 -80%、延迟 -60%
提示词缓存（KV Cache 复用）    延迟 -40%、成本 -15%
更快的模型                     延迟 -50%、成本可能增加
量化推理                       成本 -50%、质量略有下降
```

模型路由是最性价比较高的优化——将 60-70% 的简单请求分配给更小的模型。分类器本身可以是规则或一个极小的 LLM：

```python
def route_request(user_input: str) -> str:
    # 用规则快速分类
    if len(user_input) < 50 and "查询" in user_input:
        return "gpt-4o-mini"
    if "分析" in user_input or "对比" in user_input:
        return "gpt-4o"
    # 默认用小模型，复杂任务让 Agent 内部升级
    return "gpt-4o-mini"
```

### Prompt 层

```
优化手段                      影响
────────────────────────────────────────
精简 system prompt            成本 -10%、质量稳定
对话历史压缩                   成本 -20%、延迟 -30%
检索内容裁剪                   成本 -15%、质量可能提升（去掉噪音）
结构化输出（JSON mode）       成本不变、稳定性提升
```

### 缓存层

```
缓存层次           缓存对象                    典型命中率  效果
─────────────────────────────────────────────────────────────
Prompt KV Cache   相同 system prompt 的缓存     50-70%    TTFT 降低
LLM 响应缓存      相同输入 + 相同输出的缓存     10-30%    完全避免 LLM 调用
检索结果缓存      相同 query 的检索结果缓存     40-60%    避免重复检索
工具结果缓存      相同 API 调用结果缓存         20-40%    避免重复外部调用
```

LLM 响应缓存在 Agent 场景下要谨慎——同样的输入可能应该有不同的输出（因为上下文不同）。

### 架构层

```
优化手段                      影响
────────────────────────────────────────
异步非阻塞                     提高并发能力
有限状态机替代 LLM 循环        延迟 -80%、成本 -90%
规则前置                       延迟 -90%、成本零
批处理（非实时场景）           成本 -50%
```

架构层优化是最彻底的，但也是实现成本最高的。**规则前置是最被低估的优化**——能用规则判断的，不要交给 LLM。一个 `if` 语句是微秒级的，一次 LLM 调用是秒级的。

## 成本归因

没有成本归因，你就不知道钱花在哪，也不知道优化后省了多少。

### 归因维度

```
按模型:     gpt-4o: ¥45,000   gpt-4o-mini: ¥8,000
按功能:     订单查询: ¥18,000   客户咨询: ¥22,000
按用户:     企业用户: ¥35,000   个人用户: ¥18,000
按阶段:     意图识别: ¥5,000    回答生成: ¥30,000
```

每个维度给出绝对值 + 占比 + 环比变化。

### 计算公式

从 Trace 数据中计算成本，需要两个前提：
1. 每条 LLM 调用的 trace 中记录了 input/output tokens
2. 维护一个模型单价表

```python
MODEL_PRICES = {
    "gpt-4o": {
        "input": 2.50 / 1_000_000,       # $2.50/1M input tokens
        "output": 10.00 / 1_000_000      # $10.00/1M output tokens
    },
    "gpt-4o-mini": {
        "input": 0.15 / 1_000_000,
        "output": 0.60 / 1_000_000
    }
}

def calculate_llm_cost(trace: dict) -> float:
    cost = 0
    for span in trace["spans"]:
        if span["type"] == "llm_call":
            model = span.get("llm.model", "gpt-4o")
            prices = MODEL_PRICES[model]
            input_tokens = span.get("llm.input_tokens", 0)
            output_tokens = span.get("llm.output_tokens", 0)
            cost += (
                input_tokens * prices["input"]
                + output_tokens * prices["output"]
            )
    return cost
```

### 成本归因的粒度

推荐用三级粒度：

```
请求级 (每 Trace): 单个请求的成本精确值
汇总级: 按维度聚合（模型/功能/用户/时间）
趋势级: 每日/每周成本趋势，环比变化
```

## 预算管控

知道了成本归因后，就能设置预算管控。

### 三层预算模型

```
月度预算: ¥100,000
├── 模型 A (gpt-4o): ¥50,000  ← 硬上限：超过后降级到 gpt-4o-mini
├── 模型 B (gpt-4o-mini): ¥15,000
├── 向量检索: ¥8,000
├── 外部工具: ¥5,000
└── 预留: ¥22,000  ← 应对突发流量
```

### 保护机制

```
软限制 (80%): 告警推送
  通知团队，新请求自动使用更经济的模型

硬限制 (100%): 自动降级
  非核心功能降级或停止
  核心功能自动切换到更低成本的方案

每请求限制:
  单请求最多调用 10 次 LLM
  单请求最多消耗 50,000 token
  达到限制后自动终止循环
```

### 优化效果的量化

每次优化后，用 Trace 数据量化效果：

```
优化前: 每请求平均成本 ¥0.32, P95 延迟 4.2s
优化后: 每请求平均成本 ¥0.18 (-43%), P95 延迟 2.1s (-50%)
```

没有量化的优化是"自我感觉良好"。Trace 数据让你能精确回答"优化了多少"。

## 案例：一次完整的分析与优化

### 背景

客服 Agent，日活 10 万用户，日均 30 万请求。

### 现状测量

从 Trace 数据中提取基线：

```
每请求平均延迟: 4.2s (P95)
每请求平均 LLM 调用次数: 3.5
每请求平均 token: 4,500
月成本: ¥120,000

延迟构成:
  LLM 调用: 72%
  工具调用: 18%
  检索: 8%
  框架: 2%
```

### 瓶颈识别

分析 Trace 数据发现三个问题：

**问题 1**：所有请求都用 gpt-4o，但 60% 的请求很简单（查询余额、查订单状态），以 gpt-4o-mini 配合规则就能处理。

**问题 2**：Agent 的对话历史不做裁剪，每轮 step 都将全部历史塞入上下文。

```
Step 1 → Step 2 → Step 3
输入 token: 1200 → 2400 → 4800
```

第 3 步的 token 数已经是第 1 步的 4 倍，LLM 耗时同步增长。

**问题 3**：外部 API 超时设置不合理，重试次数偏多。查 Trace 发现同一个失败的 API 调用平均重试 3 次，每次 5 秒超时——一个失败请求就浪费 15 秒。

### 优化方案

```
优化 1: 模型路由
  60% 请求 → gpt-4o-mini
  预计: 成本 -50%, 延迟 -40%

优化 2: 对话历史压缩
  限制最大历史轮数 = 6 轮
  超过 6 轮的历史做摘要
  预计: 成本 -15%, 延迟 -20%

优化 3: 外部 API 超时调整
  超时从 5s → 2s
  重试次数从 3 → 1
  预计: 尾部延迟 -30%
```

### 效果量化

优化后重新测量：

```
每请求平均延迟: 1.8s (P95)    ↓ 57%
每请求平均 LLM 调用: 2.0      ↓ 43%
每请求平均 token: 2,800       ↓ 38%
月成本: ¥43,000               ↓ 64%
```

## 总结

性能分析和成本优化是一体两面——同一个 Trace 数据，既回答"哪里慢"也回答"哪里贵"。

核心方法是"三步走"：**测量（Trace 数据提取基线）→ 识别（找到占比最大的瓶颈）→ 优化（按层对症下药）**。不要跳过测量直接优化。不要优化占比小的项。

Agent 特有的延迟模型（多轮串行）决定了优化重点和传统 API 不同——控制上下文大小、减少 LLM 调用次数、规则前置——这些手段的影响远大于单纯换一个更快的模型。
